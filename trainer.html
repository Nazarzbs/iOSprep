1. Delegate pattern: екран Б повертає результат екрану А

UIKITСЕРЕДНЯ · 15 хв

Завдання. Екран A (список замовлень) відкриває екран B (вибір адреси). Користувач обирає адресу — екран B закривається, A отримує вибрану адресу і показує її. Реалізуй через delegate. Без Storyboard — усе кодом.

Що перевіряють

  • Протокол з AnyObject (інакше weak не скомпілюється)
  • weak var delegate — і чи можеш пояснити, чому weak
  • Хто кого створює і хто закриває екран B (правильно: A закриває або B через delegate повідомляє)
Підказки
  • Протокол оголошується біля екрана B («той, хто говорить», володіє контрактом).
  • A підписується: addressVC.delegate = self перед показом.
  • Retain cycle: A тримає B (навігація) → B strong delegate тримав би A → цикл.
Розвʼязок
protocol AddressPickerDelegate: AnyObject {
    func addressPicker(_ picker: AddressPickerVC, didSelect address: String)
}

final class AddressPickerVC: UIViewController {
    weak var delegate: AddressPickerDelegate?   // weak — розрив циклу

    private func didTapAddress(_ address: String) {
        delegate?.addressPicker(self, didSelect: address)
        dismiss(animated: true)
    }
}

final class OrdersVC: UIViewController, AddressPickerDelegate {
    private func showPicker() {
        let picker = AddressPickerVC()
        picker.delegate = self                    // підписка ДО показу
        present(picker, animated: true)
    }
    func addressPicker(_ picker: AddressPickerVC, didSelect address: String) {
        addressLabel.text = address
    }
}

Коментар. Перший параметр методу делегата — сам відправник (конвенція UIKit: tableView(_:didSelectRowAt:)). Альтернатива для одного колбека — closure: var onSelect: ((String) -> Void)? — згадай про це, інтервʼюери люблять порівняння.

2. UITableView з нуля повністю кодом

UIKITСЕРЕДНЯ · 15–20 хв

Завдання. Без XIB/Storyboard: екран зі списком страв (name, price). Кастомна клітинка з двома лейблами через AutoLayout. Тап по рядку — print назви. Це завдання-фільтр: перевіряє, чи писав ти UIKit руками.

Що перевіряють

  • register до dequeue; ідентифікатор клітинки
  • translatesAutoresizingMaskIntoConstraints = false
  • Констрейнти до contentView (не до cell!), конфіг у клітинці, а не в cellForRowAt
Розвʼязок
final class DishCell: UITableViewCell {
    static let reuseID = "DishCell"
    private let nameLabel = UILabel()
    private let priceLabel = UILabel()

    override init(style: UITableViewCell.CellStyle, reuseIdentifier: String?) {
        super.init(style: style, reuseIdentifier: reuseIdentifier)
        priceLabel.setContentCompressionResistancePriority(.required, for: .horizontal)
        [nameLabel, priceLabel].forEach {
            $0.translatesAutoresizingMaskIntoConstraints = false
            contentView.addSubview($0)
        }
        NSLayoutConstraint.activate([
            nameLabel.leadingAnchor.constraint(equalTo: contentView.leadingAnchor, constant: 16),
            nameLabel.centerYAnchor.constraint(equalTo: contentView.centerYAnchor),
            priceLabel.trailingAnchor.constraint(equalTo: contentView.trailingAnchor, constant: -16),
            priceLabel.centerYAnchor.constraint(equalTo: contentView.centerYAnchor),
            priceLabel.leadingAnchor.constraint(greaterThanOrEqualTo: nameLabel.trailingAnchor, constant: 8),
            contentView.heightAnchor.constraint(greaterThanOrEqualToConstant: 52),
        ])
    }
    required init?(coder: NSCoder) { fatalError() }

    func configure(name: String, price: String) {
        nameLabel.text = name; priceLabel.text = price
    }
}

final class MenuVC: UIViewController, UITableViewDataSource, UITableViewDelegate {
    private let tableView = UITableView()
    private let dishes = [("Борщ", "120 ₴"), ("Вареники", "95 ₴")]

    override func viewDidLoad() {
        super.viewDidLoad()
        tableView.register(DishCell.self, forCellReuseIdentifier: DishCell.reuseID)
        tableView.dataSource = self; tableView.delegate = self
        tableView.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(tableView)
        NSLayoutConstraint.activate([
            tableView.topAnchor.constraint(equalTo: view.topAnchor),
            tableView.bottomAnchor.constraint(equalTo: view.bottomAnchor),
            tableView.leadingAnchor.constraint(equalTo: view.leadingAnchor),
            tableView.trailingAnchor.constraint(equalTo: view.trailingAnchor),
        ])
    }
    func tableView(_ tv: UITableView, numberOfRowsInSection s: Int) -> Int { dishes.count }
    func tableView(_ tv: UITableView, cellForRowAt indexPath: IndexPath) -> UITableViewCell {
        let cell = tv.dequeueReusableCell(withIdentifier: DishCell.reuseID, for: indexPath) as! DishCell
        let dish = dishes[indexPath.row]
        cell.configure(name: dish.0, price: dish.1)
        return cell
    }
    func tableView(_ tv: UITableView, didSelectRowAt indexPath: IndexPath) {
        tv.deselectRow(at: indexPath, animated: true)
        print(dishes[indexPath.row].0)
    }
}

3. Знайди і виправ баг: «чужі» картинки при скролі

UIKITСЕРЕДНЯ · 10 хв

Завдання. Дано код клітинки, яка вантажить картинку з URL. При швидкому скролі в клітинках миготять чужі картинки. Знайди причину і виправ.

// ДАНО (з багом):
func configure(with url: URL) {
    URLSession.shared.dataTask(with: url) { data, _, _ in
        if let data, let img = UIImage(data: data) {
            DispatchQueue.main.async { self.thumbView.image = img }
        }
    }.resume()
}

Що перевіряють

  • Розуміння reuse: клітинка переюзана, а старий запит завершився пізніше
  • prepareForReuse; скасування задачі; звірка «чий це результат»
  • Бонус: strong self у closure клітинки — теж проблема
Розвʼязок
final class DishCell: UITableViewCell {
    private var currentURL: URL?
    private var task: URLSessionDataTask?

    override func prepareForReuse() {
        super.prepareForReuse()
        task?.cancel()                     // 1. скасувати старий запит
        thumbView.image = placeholder      // 2. скинути стан одразу
        currentURL = nil
    }

    func configure(with url: URL) {
        currentURL = url
        task = URLSession.shared.dataTask(with: url) { [weak self] data, _, _ in
            guard let data, let img = UIImage(data: data) else { return }
            DispatchQueue.main.async {
                guard self?.currentURL == url else { return }  // 3. це ще МОЯ картинка?
                self?.thumbView.image = img
            }
        }
        task?.resume()
    }
}

Коментар. Три рівні захисту: скасування + скидання + звірка URL. У проді це робить Kingfisher (kf.setImage сам скасовує і звіряє), але на співбесіді чекають, що ти знаєш механіку. Згадай також кеш: без нього кожен показ клітинки = новий запит.

4. Таймер зворотного відліку без витоку памʼяті

UIKITЛЕГКА · 10 хв

Завдання. Екран з лейблом «Замовлення буде готове через 05:00» — відлік щосекунди до нуля. Вимоги: контролер має нормально деінітитись при закритті екрана, відлік не має зупинятись під час скролу.

Що перевіряють

  • Timer(target: self) тримає self сильно — знаєш block-based API + [weak self]
  • invalidate: де викликати
  • RunLoop mode .common — щоб не завмирав при скролі
Розвʼязок
final class CountdownVC: UIViewController {
    private var timer: Timer?
    private var secondsLeft = 300

    override func viewDidLoad() {
        super.viewDidLoad()
        let t = Timer(timeInterval: 1, repeats: true) { [weak self] timer in
            guard let self else { timer.invalidate(); return }
            self.secondsLeft -= 1
            self.updateLabel()
            if self.secondsLeft <= 0 { timer.invalidate() }
        }
        RunLoop.main.add(t, forMode: .common)   // не завмирає при скролі
        timer = t
    }

    private func updateLabel() {
        let m = secondsLeft / 60, s = secondsLeft % 60
        label.text = String(format: "Готове через %02d:%02d", m, s)
    }

    deinit { timer?.invalidate() }   // страховка + доказ відсутності циклу
}

Коментар. Пастка follow-up питання: «а чому deinit взагалі викличеться?» — бо block-based Timer НЕ тримає self (ми взяли weak). З Timer.scheduledTimer(target: self, ...) deinit не викликався б ніколи без invalidate ззовні.

5. Збільшити тач-зону маленької кнопки

UIKITЛЕГКА · 5 хв

Завдання. Кнопка-хрестик 20×20 pt закриває банер, у неї важко влучити. Зроби тач-зону 44×44, не змінюючи візуального розміру.

Що перевіряють

  • point(inside:) vs hitTest — що перевизначати для цієї задачі
  • Знання мінімальних 44pt з HIG
Розвʼязок
final class BigTapButton: UIButton {
    var minimumHitSize = CGSize(width: 44, height: 44)

    override func point(inside point: CGPoint, with event: UIEvent?) -> Bool {
        let dx = max(0, (minimumHitSize.width  - bounds.width)  / 2)
        let dy = max(0, (minimumHitSize.height - bounds.height) / 2)
        return bounds.insetBy(dx: -dx, dy: -dy).contains(point)
    }
}

Коментар. point(inside:) — коли розширюємо зону САМОЇ вʼюхи; hitTest перевизначають у БАТЬКА, коли треба ловити тапи по дитині, що стирчить за його межі. Вмій пояснити різницю.

6. Анімоване розгортання панелі констрейнтом

UIKITЛЕГКА · 10 хв

Завдання. Панель фільтрів висотою 60 pt; по тапу розгортається до 260 з анімацією 0.3с. Реалізуй через зміну констрейнта.

Що перевіряють

  • Зберігання посилання на констрейнт
  • layoutIfNeeded усередині animate-блоку — і чому без нього не анімується
Розвʼязок
private var panelHeight: NSLayoutConstraint!
private var isExpanded = false

// setup:
panelHeight = panel.heightAnchor.constraint(equalToConstant: 60)
panelHeight.isActive = true

@objc private func togglePanel() {
    isExpanded.toggle()
    panelHeight.constant = isExpanded ? 260 : 60
    UIView.animate(withDuration: 0.3) {
        self.view.layoutIfNeeded()   // solver перераховує ЗАРАЗ,
    }                                // зміни фреймів потрапляють в анімацію
}

Коментар. Зміна constant лише помічає layout «брудним» — перерахунок стався б на наступному кадрі ПОЗА анімаційним блоком (стрибок). layoutIfNeeded усередині блоку змушує перерахунок відбутись синхронно, і UIKit анімує різницю фреймів.

7. Екран зі станами loading / loaded / error + ViewModel

SWIFTUIСЕРЕДНЯ · 20 хв

Завдання. Екран списку ресторанів: спінер при завантаженні, список при успіху, повідомлення + кнопка «Повторити» при помилці. Сервіс — async, підкинь мок з затримкою. Стан — enum. Це НАЙЧАСТІШЕ SwiftUI-завдання на співбесідах.

Що перевіряють

  • Enum стану з associated value замість трьох окремих властивостей
  • @MainActor на VM; .task для завантаження (не onAppear + Task)
  • DI сервіса через протокол — тестованість
Розвʼязок
protocol RestaurantService {
    func fetch() async throws -> [Restaurant]
}
struct MockService: RestaurantService {
    func fetch() async throws -> [Restaurant] {
        try await Task.sleep(for: .seconds(1))
        return [.init(id: "1", name: "Тарілка"), .init(id: "2", name: "Диван")]
    }
}

@MainActor @Observable
final class RestaurantsViewModel {
    enum State { case loading, loaded([Restaurant]), failed(String) }
    private(set) var state: State = .loading
    private let service: RestaurantService
    init(service: RestaurantService) { self.service = service }

    func load() async {
        state = .loading
        do    { state = .loaded(try await service.fetch()) }
        catch { state = .failed(error.localizedDescription) }
    }
}

struct RestaurantsView: View {
    @State private var vm = RestaurantsViewModel(service: MockService())

    var body: some View {
        Group {
            switch vm.state {
            case .loading:
                ProgressView("Завантаження…")
            case .loaded(let restaurants):
                List(restaurants) { Text($0.name) }
            case .failed(let message):
                VStack(spacing: 12) {
                    Text(message)
                    Button("Повторити") { Task { await vm.load() } }
                }
            }
        }
        .task { await vm.load() }   // автоскасування при зникненні
    }
}

Коментар. Якщо запитають про iOS 16 і раніше: @Observable → ObservableObject + @Published, @State vm → @StateObject. Вмій написати обидва варіанти і пояснити різницю.

8. Живий пошук із debounce

SWIFTUIСЕРЕДНЯ · 15 хв

Завдання. Поле пошуку: запит на бекенд летить не на кожен символ, а через 300мс після паузи введення; попередній незавершений запит скасовується. Без Combine — на Swift Concurrency.

Що перевіряють

  • .task(id:) — перезапуск і АВТОскасування попереднього
  • Task.sleep як debounce + перевірка скасування
  • Порожній запит не летить
Розвʼязок
struct SearchView: View {
    @State private var query = ""
    @State private var results: [Restaurant] = []

    var body: some View {
        List(results) { Text($0.name) }
            .searchable(text: $query)
            .task(id: query) {              // нова query → старий task СКАСОВАНО
                do {
                    try await Task.sleep(for: .milliseconds(300))  // debounce
                    guard !query.trimmingCharacters(in: .whitespaces).isEmpty else {
                        results = []; return
                    }
                    results = try await api.search(query)
                } catch is CancellationError {
                    // нормальний шлях: юзер продовжив друкувати
                } catch {
                    results = []
                }
            }
    }
}

Коментар. Ключовий інсайт для співбесіди: Task.sleep — «скасовний» (кидає CancellationError при cancel), тому debounce виходить безкоштовно: новий символ → .task(id:) скасовує старий таск прямо у sleep. Якщо просять Combine-варіант: debounce(for:scheduler:) + removeDuplicates + flatMap(maxPublishers: .max(1)).

9. Переюзабельний компонент із @Binding: рейтинг зірками

SWIFTUIЛЕГКА · 10 хв

Завдання. Компонент RatingView(rating: $value): пʼять зірок, тап по n-й ставить рейтинг n. Компонент не володіє станом — редагує стан власника.

Що перевіряють

  • @Binding vs @State — чому тут binding
  • $-синтаксис у власника
  • Бонус: анімація зміни
Розвʼязок
struct RatingView: View {
    @Binding var rating: Int
    var maximum = 5

    var body: some View {
        HStack(spacing: 6) {
            ForEach(1...maximum, id: \.self) { index in
                Image(systemName: index <= rating ? "star.fill" : "star")
                    .foregroundStyle(index <= rating ? .orange : .gray)
                    .onTapGesture {
                        withAnimation(.spring(duration: 0.25)) { rating = index }
                    }
            }
        }
    }
}

// Використання власником:
struct ReviewForm: View {
    @State private var rating = 0        // джерело правди ТУТ
    var body: some View {
        RatingView(rating: $rating)       // $ → Binding
    }
}

Коментар. id: \.self тут ОК — діапазон констант. Follow-up: «а якщо компоненту треба лише читати?» — тоді звичайний let rating: Int, binding лише для двостороннього звʼязку.

10. Знайди баг: стан скидається при перерендері батька

SWIFTUIСКЛАДНА · 10 хв

Завдання. Дано код (iOS 16, ObservableObject). Таймер у батьківській вʼюсі тікає щосекунди, і введений у дочірній формі текст зникає. Поясни чому і виправ.

// ДАНО (з багом):
struct ParentView: View {
    @State var seconds = 0
    var body: some View {
        VStack {
            Text("Секунд: " + String(seconds))
            FormView()
        }
        .onReceive(timer) { _ in seconds += 1 }
    }
}
struct FormView: View {
    @ObservedObject var vm = FormViewModel()   // ← баг тут
    var body: some View { TextField("Імʼя", text: $vm.name) }
}

Що перевіряють

  • Розуміння: батько перерендерився → FormView() пересоздано → @ObservedObject створив НОВИЙ VM
  • @StateObject привʼязує життя обʼєкта до identity вʼюхи, а не до struct-інстансу
Розвʼязок
struct FormView: View {
    @StateObject private var vm = FormViewModel()  // створюється ОДИН раз
    var body: some View { TextField("Імʼя", text: $vm.name) }
}

Пояснення для інтервʼюера. Щосекунди seconds змінюється → body батька → створюється новий struct FormView → @ObservedObject не володіє обʼєктом і чесно бере той, що в ініціалізаторі — новий FormViewModel() з порожнім name. @StateObject зберігає перший створений обʼєкт у сховищі SwiftUI, привʼязаному до identity вʼюхи, та ігнорує наступні autoclosure-ініціалізації. Правило: створюєш VM тут → @StateObject; отримуєш ззовні → @ObservedObject. На iOS 17+ з @Observable цей клас багів зникає: обʼєкт тримають через @State.

11. UIViewRepresentable: UITextField з фокусом і Coordinator

SWIFTUIСКЛАДНА · 20 хв

Завдання. Обгорни UITextField для SwiftUI: текст через @Binding, подія return закриває клавіатуру, властивість isFirstResponder керується ззовні. Поясни ролі makeUIView / updateUIView / Coordinator.

Що перевіряють

  • Створення ТІЛЬКИ в makeUIView; update — ідемпотентна синхронізація
  • Coordinator як delegate; звʼязок UIKit-подія → Binding
  • Захист від циклу оновлень (textField.text != text)
Розвʼязок
struct FocusableTextField: UIViewRepresentable {
    @Binding var text: String
    @Binding var isFocused: Bool
    var placeholder: String

    func makeUIView(context: Context) -> UITextField {
        let field = UITextField()
        field.placeholder = placeholder
        field.delegate = context.coordinator
        field.addTarget(context.coordinator,
                        action: #selector(Coordinator.textChanged), for: .editingChanged)
        return field
    }

    func updateUIView(_ field: UITextField, context: Context) {
        context.coordinator.parent = self          // свіжі биндінги
        if field.text != text { field.text = text }  // без циклу оновлень
        if isFocused, !field.isFirstResponder {
            DispatchQueue.main.async { field.becomeFirstResponder() }
        }
    }

    func makeCoordinator() -> Coordinator { Coordinator(self) }

    final class Coordinator: NSObject, UITextFieldDelegate {
        var parent: FocusableTextField
        init(_ parent: FocusableTextField) { self.parent = parent }

        @objc func textChanged(_ field: UITextField) {
            parent.text = field.text ?? ""          // UIKit → SwiftUI
        }
        func textFieldShouldReturn(_ field: UITextField) -> Bool {
            parent.isFocused = false
            field.resignFirstResponder()
            return true
        }
    }
}

Коментар. Це твоя робоча тема (SwiftUI в UIKit-проєкті) — скажи про це. Ролі: make — народження UIKit-обʼєкта (раз на identity); update — течія даних SwiftUI → UIKit (часто, тому дешево та ідемпотентно); Coordinator — течія назад UIKit → SwiftUI через delegate/target-action.

12. Картка, що розгортається, з анімацією і transition

SWIFTUIЛЕГКА · 10 хв

Завдання. Картка страви: тап розгортає блок з описом (плавно виїжджає згори + fade). Поясни, чому transition не працює без withAnimation.

Розвʼязок
struct DishCard: View {
    @State private var expanded = false
    let dish: Dish

    var body: some View {
        VStack(alignment: .leading, spacing: 8) {
            HStack {
                Text(dish.name).font(.headline)
                Spacer()
                Image(systemName: "chevron.down")
                    .rotationEffect(.degrees(expanded ? 180 : 0))
            }
            .contentShape(Rectangle())            // тап по всій ширині
            .onTapGesture {
                withAnimation(.spring(duration: 0.35)) { expanded.toggle() }
            }

            if expanded {
                Text(dish.details)
                    .font(.subheadline)
                    .transition(.move(edge: .top).combined(with: .opacity))
            }
        }
        .padding()
        .background(RoundedRectangle(cornerRadius: 16).fill(.background).shadow(radius: 4))
    }
}

Що перевіряють

  • AnyObject у протоколі — і чому без нього weak неможливий
  • Чому weak (напрям володіння)
  • Порівняння з closure і NotificationCenter — коли що
Розвʼязок
enum OrderStatus { case accepted, cooking, delivering, done }

protocol OrderTrackerDelegate: AnyObject {   // class-only → можна weak
    func tracker(_ tracker: OrderTracker, didUpdate status: OrderStatus)
}

final class OrderTracker {
    weak var delegate: OrderTrackerDelegate?    // weak: tracker НЕ володіє слухачем

14. Observer: міні-EventBus з відпискою

ПАТЕРНСКЛАДНА · 20 хв

Завдання. Реалізуй клас Observable<T>: subscribe повертає токен, deinit токена автоматично відписує (як AnyCancellable у Combine). Продемонструй, що після смерті підписника колбеки не викликаються і памʼять не тече.

Що перевіряють

  • Зберігання підписок без утримання підписників
  • Токен-based відписка (RAII), як у Combine
  • Потокобезпечність — хоча б згадати
Розвʼязок
final class SubscriptionToken {
    private let onCancel: () -> Void
    init(onCancel: @escaping () -> Void) { self.onCancel = onCancel }
    deinit { onCancel() }                // токен помер → відписка (RAII)
}

final class Observable<T> {
    private var observers: [UUID: (T) -> Void] = [:]
    var value: T {
        didSet { observers.values.forEach { $0(value) } }
    }
    init(_ value: T) { self.value = value }

    func subscribe(_ handler: @escaping (T) -> Void) -> SubscriptionToken {
        let id = UUID()
        observers[id] = handler
        handler(value)                    // одразу поточне значення
        return SubscriptionToken { [weak self] in
            self?.observers[id] = nil     // weak: токен не тримає observable
        }
    }
}

// Використання:
final class CartBadge {
    private var token: SubscriptionToken?
    init(cartCount: Observable<Int>) {
        token = cartCount.subscribe { [weak self] count in
            self?.render(count)          // weak: колбек не тримає badge
        }
    }   // badge помер → token помер → відписка автоматично
}

Коментар. Це спрощений AnyCancellable. Пастки, про які варто сказати вголос: (1) handler-и можуть тримати self — тому [weak self] у підписника; (2) для потокобезпечності словник observers обгорнути у serial queue або зробити клас actor-ом; (3) якщо забути зберегти токен — підписка миттєво помре (так само як у Combine).

18. Квіз: що надрукує цей код? (closures + capture)

SWIFTСЕРЕДНЯ · 10 хв

Завдання. Без запуску скажи, що виведе кожен print — і поясни чому.

// 1
var a = 10
let f1 = { print(a) }
let f2 = { [a] in print(a) }
a = 20
f1(); f2()

// 2
var fns: [() -> Int] = []
for i in 1...3 { fns.append { i } }
var j = 0
while j < 3 { j += 1; fns.append { j } }
print(fns.map { $0() })

// 3
struct S { var x = 1 }
class  C { var x = 1 }
let s = S(); let c = C()
let g = { [s, c] in print(s.x, c.x) }
// var s2 = s; s2.x = 99  — s let, тому мутуємо копію
c.x = 99
g()
Відповіді
  • 1: «20» і «10». f1 захопив ЗМІННУ за посиланням (бачить актуальне 20); f2 через capture list скопіював значення 10 у момент створення.
  • 2: [1, 2, 3, 3, 3, 3]. for на кожній ітерації створює НОВУ константу i — кожен closure тримає свою. while мутує ОДНУ змінну j — усі три closures ділять її, на момент виклику j == 3.
  • 3: «1 99». Capture list копіює: struct — копію значення (незалежна), class — копію ПОСИЛАННЯ (той самий обʼєкт, тому зміна c.x видна).

Чому це питають: перевіряє одразу value/reference семантику + capture semantics — дві найважливіші механіки мови в одному квізі.

19. Знайди 3 витоки памʼяті в коді контролера

SWIFTСКЛАДНА · 15 хв

Завдання. У цьому контролері три причини, чому deinit ніколи не викличеться. Знайди всі й виправ.

final class ProfileVC: UIViewController {
    var onUpdate: (() -> Void)?
    var timer: Timer?

    override func viewDidLoad() {
        super.viewDidLoad()
        onUpdate = { self.refresh() }                          // ?
        timer = Timer.scheduledTimer(timeInterval: 10, target: self,
            selector: #selector(tick), userInfo: nil, repeats: true) // ?
        NotificationCenter.default.addObserver(forName: .userChanged,
            object: nil, queue: .main) { _ in self.refresh() }  // ?
    }
}
Розвʼязок
  • Витік 1 — closure-властивість: self тримає onUpdate, onUpdate тримає self. Фікс: onUpdate = { [weak self] in self?.refresh() }.
  • Витік 2 — Timer з target: RunLoop → Timer → strong target (self). Навіть weak var timer не рятує — тримає САМ таймер. Фікс: block-based API + [weak self] + invalidate у viewDidDisappear/deinit.
  • Витік 3 — NotificationCenter closure-API: центр тримає блок, блок тримає self; до iOS 9 навіть повернений токен треба знімати. Фікс: [weak self] у блоці + зберігати токен і removeObserver(token).
// Виправлено:
onUpdate = { [weak self] in self?.refresh() }

timer = Timer.scheduledTimer(withTimeInterval: 10, repeats: true) { [weak self] _ in
    self?.tick()
}

token = NotificationCenter.default.addObserver(forName: .userChanged,
    object: nil, queue: .main) { [weak self] _ in self?.refresh() }

deinit {
    timer?.invalidate()
    if let token { NotificationCenter.default.removeObserver(token) }
    print("ProfileVC deinit ✓")
}

Бонус-відповідь: «як я б це знайшов у реальному проєкті» — print у deinit, Debug Memory Graph (хто тримає обʼєкт), Instruments Leaks.

20. Потокобезпечний лічильник: GCD і actor

SWIFTСЕРЕДНЯ · 15 хв

Завдання. 100 потоків одночасно інкрементують лічильник 1000 разів. Наївний var count += 1 дає число менше за 100 000 (data race). Зроби потокобезпечно двома способами: GCD (читання паралельне, запис ексклюзивний) і actor.

Розвʼязок
// Спосіб 1: concurrent queue + barrier (readers-writers)
final class SafeCounter {
    private var count = 0
    private let queue = DispatchQueue(label: "counter", attributes: .concurrent)

    func increment() {
        queue.async(flags: .barrier) { self.count += 1 }  // запис — ексклюзивний
    }
    var value: Int {
        queue.sync { count }                              // читання — паралельні
    }
}

// Спосіб 2: actor — компілятор гарантує серіалізацію
actor Counter {
    private(set) var count = 0
    func increment() { count += 1 }
}

// Використання actor:
let counter = Counter()
await withTaskGroup(of: Void.self) { group in
    for _ in 1...100 {
        group.addTask {
            for _ in 1...1000 { await counter.increment() }
        }
    }
}
print(await counter.count)   // рівно 100000

Для розмови. Чому barrier, а не serial queue: читань зазвичай набагато більше — вони йдуть паралельно, блокується лише запис. Чому actor кращий у 2026: помилку неможливо зробити — компілятор змусить await і не дасть синхронно залізти в стан. Згадай третій варіант — NSLock/os_unfair_lock для гарячих шляхів (найшвидший, але ручний).

    let id: Int

func fetch<T: Decodable>(_ type: T.Type, from url: URL) async throws -> T {
    let (data, response) = try await URLSession.shared.data(from: url)
    guard let http = response as? HTTPURLResponse,
          (200..<300).contains(http.statusCode) else {
        throw APIError.badStatus((response as? HTTPURLResponse)?.statusCode ?? -1)
    }
    let decoder = JSONDecoder()
    decoder.keyDecodingStrategy = .convertFromSnakeCase
    do  { return try decoder.decode(T.self, from: data) }
    catch { throw APIError.decoding(error) }
}

22. Алгоритмічна розминка: 4 класики за 20 хвилин

SWIFTЛЕГКА · 20 хв

Завдання. Без стандартних «читерських» функцій там, де просять вручну: (1) розверни масив in-place; (2) перший символ, що не повторюється; (3) знайди всі дублікати; (4) чи є два числа з сумою target. Назви складність кожного.

Розвʼязок
// 1. Reverse in-place — O(n), two pointers
func reverseInPlace<T>(_ a: inout [T]) {
    var i = 0, j = a.count - 1
    while i < j { a.swapAt(i, j); i += 1; j -= 1 }
}

// 2. Перший неповторюваний символ — O(n), два проходи
func firstUnique(_ s: String) -> Character? {
    var counts: [Character: Int] = [:]
    for ch in s { counts[ch, default: 0] += 1 }
    return s.first { counts[$0] == 1 }
}

// 3. Дублікати — O(n) з Set
func duplicates(_ nums: [Int]) -> [Int] {
    var seen = Set<Int>(), dups = Set<Int>()
    for n in nums {
        if !seen.insert(n).inserted { dups.insert(n) }
    }
    return Array(dups)
}

// 4. Two Sum — O(n): «яке число мені потрібне?»
func hasPair(_ nums: [Int], target: Int) -> Bool {
    var seen = Set<Int>()
    for n in nums {
        if seen.contains(target - n) { return true }
        seen.insert(n)
    }
    return false
}

Порада. Для iOS-позицій middle-рівня в Україні цього рівня алгоритмів зазвичай достатньо (легкі LeetCode). Проговорюй складність ДО написання коду — «зроблю за O(n) через Set» звучить сильно. Патерн 4-ї задачі (шукаю доповнення в Set) — найчастіша алгоритмічна задача iOS-співбесід.
