Value vs Reference types, struct vs class

🧵 P0
Розгорнути тему
Що це і як працює

Value-типи (struct, enum, tuple) при присвоєнні та передачі у функцію копіюються — кожна змінна володіє власним незалежним значенням. Reference-типи (class, closure, actor) передаються за посиланням: кілька змінних вказують на один обʼєкт у купі (heap), і зміна через одне посилання видна всім.

Стандартні колекції (Array, String, Dictionary, Set) — value-типи з оптимізацією Copy-on-Write: фізичне копіювання буфера відбувається лише в момент першої мутації, а не при присвоєнні. Тому передати масив на 100 000 елементів у функцію — дешево, доки функція його не змінює.

Яку проблему вирішує

Головна проблема, яку вирішують value-типи — спільний мутабельний стан (shared mutable state): коли два шматки коду тримають один обʼєкт і несподівано змінюють його один одному «під ногами». З value-типами про шматок коду можна міркувати локально: ніхто ззовні твою копію не змінить. Це також безкоштовна потокобезпечність — копії незалежні.

Де використовується на практиці
  • Моделі даних і DTO з бекенда — майже завжди struct (у твоєму проєкті LocationAddress, моделі замовлення тощо). Codable + struct — стандартна пара.
  • SwiftUI: кожна View — struct. Уся ідеологія фреймворку побудована на дешевих value-«описах» інтерфейсу.
  • class — коли потрібна ідентичність та спільний стан: мережевий менеджер, кеш, сесія користувача, ViewModel (ObservableObject), усе успадковане від UIKit (UIViewController, UIView — класи, бо їм потрібна саме ідентичність конкретного обʼєкта на екрані).
  • enum — для станів і скінченних варіантів: стан екрана, тип оплати, маршрути дип-лінків.
Приклад
struct Point { var x = 0 }
var a = Point(); var b = a
b.x = 10                    // a.x == 0 — незалежна копія

final class Counter { var value = 0 }
let c1 = Counter(); let c2 = c1
c2.value = 10               // c1.value == 10 — один обʼєкт
// let для класу фіксує ПОСИЛАННЯ, не вміст!
Пастки та типові помилки
  • Struct, який містить class-властивість: копіюється struct, але посилання всередині вказує на той самий обʼєкт — «value-семантика» ламається.
  • let класу не робить обʼєкт незмінним — лише забороняє переприсвоїти посилання. let структури — заморожує все.
  • mutating-метод структури неможливо викликати на let-змінній — і це фіча, а не обмеження.
  • Дуже великий struct, який часто копіюється у гарячому коді, може програти класу за перформансом — міряй Instruments, а не вгадуй.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Що виведе код, де struct присвоєно у дві змінні і одну змінили?
  • Чому Apple рекомендує struct за замовчуванням?
  • Як COW впливає на продуктивність передачі великого масиву у функцію? Як перевірити, що копія відбулась?
  • Наведи приклад зі свого проєкту, де value-семантика врятувала від бага (або де потрібен був саме class).

Optionals

🧵 P0
Розгорнути тему
Що це і як працює

Optional<T> — це звичайний enum з двома кейсами: .some(T) і .none. Синтаксис T? — лише цукор. Optional вбудовує «відсутність значення» у систему типів: компілятор змушує явно обробити кейс nil ще до запуску.

Способи роботи: if let / guard let (безпечне розгортання), оператор ?? (значення за замовчуванням), optional chaining user?.address?.city (ланцюг обривається на першому nil), map/flatMap на Optional, force unwrap ! (крешить на nil).

Яку проблему вирішує

Вирішує проблему «мільярдної помилки» — null pointer. В Objective-C виклик методу на nil мовчки нічого не робив (приховані баги), у Java NPE валив програму в рантаймі. Swift переносить цю помилку на етап компіляції: тип String і String? — різні типи, і забути перевірку неможливо.

Де використовується на практиці
  • Відповіді API: поля, яких може не бути у JSON — var promoCode: String? у моделі замовлення. Codable сам ставить nil для відсутніх ключів, якщо поле опціональне.
  • UI-стан «ще не завантажено»: var user: User? = nil до приходу відповіді.
  • Повернення з пошуку: dictionary[key], array.first, Int("abc") — усе повертає Optional, бо результату може не бути.
  • guard let — стандарт на початку методу: перевірили вхідні дані й далі працюємо з не-опціоналами (early exit).
Приклад
func handle(response: Data?) {
    guard let data = response else {
        showError(); return   // early exit — далі data гарантовано є
    }
    let order = try? decoder.decode(Order.self, from: data)
    // try? повертає Order? — помилка перетворилась на nil
    label.text = order?.address?.city ?? "—"
}
Пастки та типові помилки
  • Force unwrap (!) у продакшні — креш, який знайде користувач. Легітимні винятки: IBOutlet (система гарантує звʼязування), ресурси в бандлі, які точно існують.
  • try? + функція, що повертає Optional = подвійний optional T?? — легко заплутатись.
  • if let x = x всередині кожного if — «піраміда»; краще один guard з комою: guard let a, let b else.
  • map vs flatMap на Optional: map загортає результат ще раз (може дати T??), flatMap сплющує.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Що таке Optional під капотом? Напиши його оголошення як enum.
  • Чим map на Optional відрізняється від flatMap?
  • Чому force unwrap — червоний прапорець і які винятки прийнятні?
  • guard let vs if let — коли що і чому guard покращує читабельність?

Properties: stored, computed, lazy, observers, wrappers

🧵 P1
Розгорнути тему
Що це і як працює

Stored — зберігає значення в памʼяті обʼєкта. Computed — не зберігає нічого, обчислюється при кожному звертанні (get, опційно set). lazy — stored, який ініціалізується при першому звертанні, а не при створенні обʼєкта. Observers willSet/didSet — хуки навколо запису у stored property.

Property wrapper — тип із var wrappedValue, який інкапсулює логіку зберігання; @State, @Published, @AppStorage — усе це property wrappers. projectedValue (доступ через $) дає додаткове представлення — наприклад, Binding у SwiftUI.

Яку проблему вирішує

lazy вирішує проблему дорогої ініціалізації, яка може не знадобитись (важкий DateFormatter, обчислення на великих даних). Observers — проблему «тримати щось синхронним зі значенням» без ручних викликів (оновити UI при зміні моделі). Wrappers — проблему копіпасти навколо зберігання (та сама логіка UserDefaults для 20 налаштувань — один wrapper замість 20 однакових get/set).

Де використовується на практиці
  • lazy var formatter: DateFormatter — класика: формуттери дорогі, створюємо один раз при першому використанні.
  • didSet на властивості ViewModel/клітинки: var order: Order? { didSet { updateUI() } } — типовий патерн конфігурації UITableViewCell.
  • @UserDefault("onboarding_shown") var shown: Bool — свій wrapper для налаштувань (поширене питання «напишіть property wrapper»).
  • Computed для похідних даних: var fullPrice: Decimal { items.reduce(0) { $0 + $1.price } } — завжди актуальне, немає ризику розсинхрону.
Приклад
@propertyWrapper struct UserDefault<T> {
    let key: String; let defaultValue: T
    var wrappedValue: T {
        get { UserDefaults.standard.object(forKey: key) as? T ?? defaultValue }
        set { UserDefaults.standard.set(newValue, forKey: key) }
    }
}
struct Settings {
    @UserDefault(key: "isDark", defaultValue: false) static var isDark: Bool
}
Пастки та типові помилки
  • lazy НЕ потокобезпечний: два потоки при першому звертанні можуть ініціалізувати двічі. І lazy — тільки var.
  • willSet/didSet не викликаються під час init (пряме присвоєння у сховище) — частий сюрприз.
  • Computed property обчислюється при КОЖНОМУ звертанні — дорогий обрахунок у ньому в циклі = просідання.
  • didSet, який присвоює цю ж властивість — рекурсії не буде (спрацює лише раз), але код заплутаний.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Чим lazy var відрізняється від computed property за кількістю обчислень і памʼяттю?
  • Чому didSet не викликається з ініціалізатора і як усе-таки виконати логіку?
  • Напиши property wrapper @Clamped(0...100) для відсотків.
  • Що таке projectedValue і що повертає $ у @Published/@State?

Closures і capture semantics

🧵 P0
Розгорнути тему
Що це і як працює

Closure — блок коду першого класу, який можна зберігати і передавати. Це reference-тип: він живе в купі й захоплює (capture) змінні з контексту створення. var захоплюється за посиланням — closure бачить актуальне значення і може його змінювати. Capture list ([x], [weak self]) — копіює значення/послаблює посилання В МОМЕНТ СТВОРЕННЯ closure.

@escaping — closure переживає виклик функції (зберігається, викликається пізніше — completion handlers). Non-escaping (за замовчуванням) виконується до повернення функції, тому не вимагає self і не створює циклів.

Яку проблему вирішує

Closures вирішують проблему «передати поведінку як дані»: колбеки завершення асинхронних операцій, конфігурація (map/filter), відкладене виконання. Capture list вирішує дві проблеми: retain cycle (closure тримає self, а self тримає closure) і «захоплення останнього значення» у циклах.

Де використовується на практиці
  • Completion handlers мережевих запитів (до async/await, і досі в legacy): fetchOrders { [weak self] result in ... }.
  • Колбеки UI: кнопка в клітинці повідомляє контролер через var onTap: (() -> Void)? — стандартний патерн у твоєму UIKit-коді.
  • Функціональні операції: orders.filter { $0.isActive }.map(\.id).
  • DispatchQueue.main.async { }, анімації UIView.animate — усюди closures.
Приклад
var handlers: [() -> Void] = []
for i in 1...3 {
    handlers.append { print(i) }          // 1 2 3 — for створює нову константу i
}
var x = 0
let a = { print(x) }                      // захоплення за посиланням
let b = { [x] in print(x) }               // копія на момент створення
x = 99
a() // 99        b() // 0
Пастки та типові помилки
  • Retain cycle: self.completion = { self.doSomething() } — self тримає closure, closure тримає self. Розрив: [weak self].
  • [weak self] робить self опціональним усередині — guard let self else { return } на початку.
  • unowned self — креш, якщо closure викличеться після смерті self. Тільки коли життя closure гарантовано коротше за self.
  • [weak self] не потрібен там, де closure не зберігається надовго: разовий DispatchQueue.main.async лише відкладе звільнення на мить — це не цикл.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Що виведе код вище і чому a() та b() дають різні результати?
  • Чому @escaping вимагає явного self? Що перевіряє компілятор?
  • Namaлюй retain cycle через closure і два способи розірвати.
  • Коли [weak self] зайвий і лише додає шуму?

ARC, memory management, retain cycles

🧵 P0
Розгорнути тему
Що це і як працює

ARC (Automatic Reference Counting) — компілятор на етапі збірки вставляє retain/release виклики. Кожен обʼєкт має лічильник сильних посилань; коли він падає до нуля — deinit і звільнення. На відміну від garbage collection: детерміновано (обʼєкт помирає одразу), без пауз збирача, але не вміє розривати цикли — це відповідальність розробника.

weak — не збільшує лічильник, автоматично стає nil після смерті обʼєкта (тому завжди optional var). unowned — не збільшує лічильник, але НЕ обнуляється: звертання після смерті = креш. strong — за замовчуванням.

Яку проблему вирішує

Вирішує проблему ручного керування памʼяттю (retain/release руками в старому Obj-C — джерело крешів і ліків). Retain cycle — головна ціна: два обʼєкти тримають один одного сильно, лічильники ніколи не впадуть до нуля, память тече. У великому застосунку типовий симптом — контролер не деінітиться і продовжує слухати нотифікації/оновлювати дані.

Де використовується на практиці
  • delegate завжди weak var delegate: SomeDelegate? — інакше цикл: VC тримає tableView, tableView тримає delegate (VC).
  • closure + self — див. тему closures; це 90% реальних ліків.
  • Timer.scheduledTimer(target: self) — RunLoop тримає таймер, таймер СИЛЬНО тримає target. Не інвалідував таймер — VC живе вічно.
  • Батько-дитина: батько тримає дітей strong, дитина батька — weak (view ієрархії, координатори).
Приклад
final class OrderVC: UIViewController {
    var timer: Timer?
    override func viewDidLoad() {
        super.viewDidLoad()
        timer = Timer.scheduledTimer(withTimeInterval: 5, repeats: true) { [weak self] _ in
            self?.refreshStatus()     // block-based API + weak — циклу немає
        }
    }
    deinit { timer?.invalidate(); print("OrderVC deinit") } // лог = перевірка ліків
}
Пастки та типові помилки
  • weak тільки для класів (reference-типів) — на struct не повісиш.
  • NotificationCenter.addObserver(forName:...) з closure повертає токен — його треба зберігати і знімати; сам closure легко захоплює self.
  • unowned виглядає «чистіше» (без optional), але один неправильний прогноз часу життя = креш у проді. Сумніваєшся — weak.
  • Пошук ліків: Debug Memory Graph (кнопка в Xcode) показує, ХТО тримає обʼєкт; Instruments → Leaks/Allocations; найпростіше — print у deinit.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Як ARC відрізняється від garbage collection — плюси й мінуси?
  • Таблиця strong/weak/unowned: чи тримає, чи optional, що при звертанні до мертвого?
  • Чому delegate — weak? Що станеться, якщо зробити strong?
  • Як ти шукав/шукав би memory leak у реальному проєкті — покроково?

GCD і класична багатопотоковість

🧵 P0
Розгорнути тему
Що це і як працює

GCD (Grand Central Dispatch) — API черг задач поверх пулу потоків. Serial черга виконує задачі по одній у порядку додавання; concurrent — кілька одночасно. main — serial черга головного потоку (весь UI). Global-черги — concurrent, з рівнями QoS: userInteractive > userInitiated > utility > background — система розподіляє ресурси за пріоритетом.

async — «поклади задачу і не чекай»; sync — «поклади і заблокуйся до завершення». DispatchGroup — дочекатися групи задач; barrier — задача-«шлюз» у concurrent черзі: чекає всі попередні, виконується сама, потім пускає наступні (патерн readers-writers).

Яку проблему вирішує

Вирішує дві проблеми: (1) не блокувати main thread важкою роботою — інакше фризи UI і креш від watchdog; (2) керувати паралелізмом без ручного створення потоків. Race condition — коли результат залежить від порядку виконання потоків; data race — одночасний неатомарний доступ до памʼяті (мінімум один запис) — це UB, ловиться Thread Sanitizer.

Де використовується на практиці
  • Класичний сендвіч: DispatchQueue.global().async { важка робота } → DispatchQueue.main.async { оновити UI }.
  • Декодування великого JSON, ресайз картинок, робота з диском — усе поза main.
  • DispatchGroup: зібрати 3 запити (меню + промо + профіль) і показати екран, коли ВСІ готові — твій реальний кейс на home page.
  • Serial черга як памʼятка синхронізації: усі звертання до спільного кеша через одну serial чергу = потокобезпечність без локів.
Приклад
let group = DispatchGroup()
var menu: Menu?; var promos: [Promo] = []
group.enter(); api.fetchMenu   { menu = $0;   group.leave() }
group.enter(); api.fetchPromos { promos = $0; group.leave() }
group.notify(queue: .main) { self.render(menu, promos) }

// Потокобезпечний доступ: concurrent + barrier
queue.async(flags: .barrier) { self.cache[key] = value }  // запис — ексклюзивно
queue.sync { return self.cache[key] }                     // читання — паралельно
Пастки та типові помилки
  • DispatchQueue.main.sync { } з main thread = миттєвий deadlock: main чекає задачу, яка чекає main. Питання №1 на співбесідах.
  • sync на будь-якій serial черзі зсередини неї самої — той самий deadlock.
  • Забутий group.leave() (наприклад, у гілці помилки) — notify не спрацює ніколи.
  • GCD не скасовує задачі, що вже виконуються; DispatchWorkItem.cancel() — лише прапорець isCancelled. Потрібне справжнє скасування/залежності — OperationQueue або Swift Concurrency.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Що станеться при DispatchQueue.main.sync {} з main thread — і чому саме?
  • Serial vs concurrent, sync vs async — чотири комбінації, поясни кожну.
  • Як завантажити 3 ресурси паралельно і оновити UI, коли всі готові — GCD і async/await варіанти?

Swift Concurrency: async/await, actors, Sendable

🧵 P0
Розгорнути тему
Що це і як працює

async/await — синтаксис для асинхронності без колбеків: await позначає suspension point, де функція може «заснути», звільнивши потік для іншої роботи (кооперативний пул потоків ≈ кількість ядер; потоки не блокуються). Task { } — міст із синхронного світу в асинхронний; structured concurrency (async let, TaskGroup) — дочірні задачі не переживають батьківську область, скасування поширюється автоматично.

actor — reference-тип, який серіалізує доступ до свого стану: одночасно всередині виконується лише один виклик, data race неможливий за визначенням. @MainActor — глобальний актор головного потоку: вішається на клас/метод, і компілятор гарантує виконання на main. Sendable — маркер «тип безпечно передавати між потоками»; у Swift 6 strict concurrency компілятор перевіряє це на етапі збірки.

Яку проблему вирішує

Вирішує: (1) callback hell і забуті completion(...) у гілках помилок — лінійний код з try/await читається як синхронний; (2) data races — actors та Sendable переносять помилки конкурентності з рантайму (найважчі в дебагу баги) у compile time; (3) скасування — Task.cancel() поширюється деревом задач, GCD такого не має.

Де використовується на практиці
  • Мережевий шар: let (data, response) = try await session.data(for: request) — сучасний стандарт, у тебе на роботі так.
  • Паралельні незалежні запити: async let menu = api.menu(); async let promos = api.promos(); let page = try await (menu, promos).
  • @MainActor на ViewModel — усі @Published/стан оновлюються на main без ручних DispatchQueue.main.async.
  • AsyncStream — обгортка над делегатами/нотифікаціями: стрім оновлень геолокації, статусів замовлення (твої Live Activities поруч із цим).
  • withCheckedThrowingContinuation — обгортання старого callback-API (Firebase, сторонні SDK) в async.
Приклад
func loadImages(urls: [URL]) async throws -> [UIImage] {
    try await withThrowingTaskGroup(of: UIImage.self) { group in
        for url in urls {
            group.addTask { try await self.download(url) }   // паралельно
        }
        var result: [UIImage] = []
        for try await image in group { result.append(image) }
        return result
    }
}

actor ImageCache {
    private var storage: [URL: UIImage] = [:]
    func image(for url: URL) -> UIImage? { storage[url] }   // доступ серіалізовано
    func insert(_ img: UIImage, for url: URL) { storage[url] = img }
}
Пастки та типові помилки
  • Actor reentrancy: під час await усередині actor-метода інший виклик може зайти і змінити стан — інваріанти перевіряй ПІСЛЯ await заново.
  • Task { } успадковує актор-контекст місця створення; Task.detached — ні (і пріоритет теж). Detached — майже завжди неправильний вибір.
  • Скасування кооперативне: якщо код не перевіряє Task.isCancelled / не викликає try Task.checkCancellation() — він виконається до кінця попри cancel().
  • async let без await результату — задача все одно виконається і буде скасована при виході з області; try await обовʼязково для throwing.
  • Блокувати потік з async-контексту (semaphore.wait, sleep) — забирає потік з кооперативного пулу, ризик starvation.
Контрольні питання — відповів уголос за 1–2 хв → став ✓

Protocols і Protocol-Oriented Programming

🧵 P0
Розгорнути тему
Що це і як працює

Протокол — контракт вимог (методи, властивості, associated types). Protocol extension дає реалізації за замовчуванням — так будується POP: спільна поведінка через композицію протоколів замість ієрархії класів. Струкrypи можуть підкорятися багатьом протоколам — «множинне успадкування поведінки» без класів.

some Protocol (opaque type) — конкретний тип, відомий компілятору, але прихований від читача: статичний dispatch, нуль накладних витрат. any Protocol (existential) — «коробка», у якій може лежати будь-який тип, що підкоряється: динамічний dispatch, боксинг, повільніше. Правило: some де можливо, any де потрібна справжня гетерогенність.

Яку проблему вирішує

Вирішує жорсткість класового успадкування: один базовий клас, глибокі ієрархії, «god object» нагорі. І головне для практики — розвʼязування залежностей: код залежить від абстракції (протоколу), а не конкретного типу, що робить його тестованим (підміна мока) і замінюваним (переїхали з одного SDK на інший — переписали одну реалізацію).

Де використовується на практиці
  • DI + тестування — головний практичний кейс: protocol OrderServiceProtocol { func fetchOrders() async throws -> [Order] }; ViewModel приймає протокол, у тестах — MockOrderService. Це закриває твій геп у тестуванні.
  • Твій мережевий шар: типізовані запити — це протокол Endpoint з associated type відповіді.
  • Delegate-патерн UIKit — протоколи (UITableViewDataSource).
  • Стандартна бібліотека: Equatable, Hashable, Codable, Sequence — усе протоколи; кастомний тип у Set вимагає Hashable.
Приклад
protocol OrderService {
    func fetchOrders() async throws -> [Order]
}
struct APIOrderService: OrderService { /* реальні запити */ }
struct MockOrderService: OrderService {
    var stub: [Order] = []
    func fetchOrders() async throws -> [Order] { stub }
}
// Пастка static dispatch у extension:
protocol Greeter { }               // метод НЕ оголошений у протоколі
extension Greeter { func hi() -> String { "proto" } }
struct Kyiv: Greeter { func hi() -> String { "kyiv" } }
let g: Greeter = Kyiv()
g.hi()   // "proto"! — метод не в вимогах → static dispatch за типом змінної
Пастки та типові помилки
  • Класика-пастка (у коді вище): метод, оголошений ТІЛЬКИ в extension протоколу, диспатчиться статично за типом змінної. Додай метод у вимоги протоколу — стане динамічним і поверне "kyiv".
  • any Protocol має ціну: existential container, непрямі виклики. У гарячих місцях і колекціях краще generic constraint.
  • Протокол з associated type до Swift 5.7 не можна було використовувати як тип — тільки як constraint; тепер any/some це помʼякшили (primary associated types).
  • Протокол для делегата має бути AnyObject (class-only) — інакше weak не повісиш.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Пастка: метод тільки в extension протоколу — що викличеться через змінну типу протоколу і чому?
  • some vs any: різниця в dispatch, перформансі, можливостях? Чому SwiftUI повертає some View?
  • Як протоколи роблять ViewModel тестованим — покажи на прикладі мока мережі.
  • POP vs успадкування: наведи приклад рефакторингу ієрархії на протоколи.

Generics

🧵 P1
Розгорнути тему
Що це і як працює

Generics — параметризація коду типами: func max<T: Comparable>(_ a: T, _ b: T) -> T працює для будь-якого порівнюваного типу, зберігаючи типобезпеку (на виході саме T, не Any). Constraints (T: Codable, where Element == String) звужують можливості до потрібних. Компілятор спеціалізує generic-код під конкретні типи — на виході швидкий мономорфний код без боксингу.

Conditional conformance: «масив Equatable-елементів сам Equatable» — extension Array: Equatable where Element: Equatable. Primary associated types (Swift 5.7): some Collection<String>.

Яку проблему вирішує

Вирішує дилему «дублювати код під кожен тип чи втратити типи через Any». До generics контейнери або писались під один тип, або повертали Any з кастами в рантаймі (і крешами). Generics дають одночасно перевикористання І compile-time безпеку І перформанс.

Де використовується на практиці
  • Твій мережевий шар — найкращий приклад на співбесіді: func request<T: Decodable>(_ endpoint: Endpoint) async throws -> T — один метод декодує будь-яку модель.
  • Уся стандартна бібліотека: Array<Element>, Dictionary<Key, Value>, Optional<Wrapped>, Result<Success, Failure>.
  • Generic-репозиторій/сховище: save<T: Encodable>(_ value: T, key: String).
  • SwiftUI повсюдно: ForEach<Data, ID, Content>, Button<Label>.
Приклад
struct Endpoint<Response: Decodable> {
    let path: String
    let method: HTTPMethod
}
extension Endpoint {
    static var orders: Endpoint<[Order]> { .init(path: "/orders", method: .get) }
}
func request<R>(_ endpoint: Endpoint<R>) async throws -> R {
    let (data, _) = try await session.data(for: makeURLRequest(endpoint))
    return try decoder.decode(R.self, from: data)
}
let orders = try await request(.orders)   // тип виведено: [Order]
Пастки та типові помилки
  • Generic-параметр фіксується на момент виклику — не можна в один [T] покласти різні типи (для цього existential any).
  • Занадто загальні констрейнти → нечитабельні сигнатури; іноді простіше два перевантаження.
  • Стирання типів: коли треба зберегти «будь-який Endpoint» у масиві — потрібен type erasure або any.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Напиши generic-функцію пошуку максимального елемента з constraint.
  • Чим generic параметр відрізняється від any Protocol за перформансом (спеціалізація vs existential)?
  • Розкажи, як влаштований типізований запит у вашому мережевому шарі.
  • Що таке conditional conformance? Приклад зі стандартної бібліотеки.

Enums, pattern matching, error handling

🧵 P1
Розгорнути тему
Що це і як працює

Swift enum — повноцінний тип: методи, computed properties, generic-параметри. Associated values прикріплюють дані до кейса — case loaded([Order]), case failed(APIError); switch змушує обробити ВСІ кейси (exhaustiveness) — додав кейс, компілятор покаже всі місця, які треба оновити.

Error handling: функція позначається throws, помилка кидається throw, ловиться do/catch. try? перетворює помилку на nil, try! — на креш. rethrows — функція кидає, лише якщо кидає передане їй closure (map, filter). defer — блок, що виконається при БУДЬ-ЯКОМУ виході з області (кілька defer — у зворотному порядку).

Яку проблему вирішує

Enum з associated values вирішує проблему неможливих станів: замість isLoading: Bool + error: Error? + data: [Order]? (де можливі безглузді комбінації «і завантажується, і помилка») — один enum, де стан рівно один. Це прибирає цілий клас багів ще на етапі моделювання.

Де використовується на практиці
  • Стан екрана: enum ScreenState { case idle, loading, loaded([Order]), failed(String) } — і SwiftUI/UIKit рендерить switch-ом.
  • Помилки API: enum APIError: Error { case unauthorized, server(code: Int), decoding } — твій мережевий шар з централізованою обробкою помилок робить саме це.
  • Маршрути дип-лінків: enum DeepLink { case order(id: String), restaurant(id: String), promo(code: String) } — парсинг URL в enum, далі типобезпечна навігація (твій робочий кейс).
  • Result<T, E> — там, де async/await недоступний: колбеки, збереження результату для повторної обробки.
Приклад
enum ScreenState {
    case loading
    case loaded([Order])
    case failed(message: String)
}
func render(_ state: ScreenState) {
    switch state {
    case .loading: showSpinner()
    case .loaded(let orders) where orders.isEmpty: showEmptyView()
    case .loaded(let orders): show(orders)
    case .failed(let message): showError(message)
    }
}
if case .loaded(let orders) = state { print(orders.count) } // один кейс без switch
Пастки та типові помилки
  • @unknown default для системних enum (нові кейси в нових iOS — код не зламається, але warning підкаже).
  • Indirect enum для рекурсивних структур (дерево категорій меню) — без indirect не скомпілюється (нескінченний розмір).
  • try? ковтає ПРИЧИНУ помилки — для дебагу продакшн-багів краще do/catch з логуванням.
  • raw value vs associated value — взаємовиключні; rawValue зручний для парсингу з бекенда (enum Status: String, Codable).
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Змоделюй стан екрана через enum — чому це краще за три окремі властивості?
  • Що робить rethrows і де він у стандартній бібліотеці (map)?
  • Порядок виконання трьох defer у функції?
  • Як ти парсиш дип-лінки — розкажи про enum маршрутів.

Method dispatch: static, table, message

🧵 P2
Розгорнути тему
Що це і як працює

Три механізми виклику методу. Static (direct) — адреса відома на компіляції: усі методи struct/enum, final-методи, private, методи в extension. Найшвидший, дозволяє інлайнінг. Table (vtable/witness table) — адреса береться з таблиці за індексом у рантаймі: звичайні методи класів (можливий override), вимоги протоколів. Message dispatch — пошук селектора через Objective-C runtime: @objc dynamic методи; найповільніший, але дає магію — KVO, method swizzling, target-action.

Яку проблему вирішує

Розуміння dispatch пояснює «дивну» поведінку (пастка з extension протоколу), чому final/private прискорюють код (компілятор девіртуалізує та інлайнить), і чому KVO вимагає @objc dynamic (KVO в рантаймі підміняє клас і перехоплює сеттер — можливо лише з message dispatch).

Де використовується на практиці
  • final class / final func на класах, які не успадковуються — безкоштовна оптимізація і явний намір.
  • @objc dynamic — коли потрібен KVO (спостереження за AVPlayer.status), UIAppearance.
  • Розуміння пастки protocol extension (див. тему протоколів) — це питання саме про dispatch.
Приклад
class Animal {
    func sound() { }          // table dispatch — можна override
    final func id() { }       // static — override заборонено, інлайниться
    @objc dynamic var name = "" // message — видно KVO і swizzling
}
extension Animal {
    func walk() { }           // static! override у нащадку НЕ спрацює як очікуєш
}
Пастки та типові помилки
  • Метод у extension класу — static dispatch: «перевизначення» у сабкласі не викличеться через посилання на базовий тип.
  • Той самий ефект у protocol extension без оголошення методу в протоколі.
  • dynamic без потреби — втрата оптимізацій на рівному місці.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Чому KVO вимагає @objc dynamic — що робить рантайм?
  • Який dispatch у методу з extension класу і яку проблему це створює?
  • Як final допомагає компілятору оптимізувати?

Колекції та функціональні операції

🧵 P1
Розгорнути тему
Що це і як працює

map — перетворити кожен елемент (O(n)); compactMap — map + викинути nil; flatMap — map + сплющити вкладені колекції; filter — відібрати; reduce — згорнути в одне значення; sorted — O(n log n). Ланцюги операцій читаються як опис «що зробити», а не «як ітерувати».

Sequence — можна ітерувати (можливо, одноразово і нескінченно); Collection — багаторазова, з індексами і count. .lazy відкладає обчислення: ланцюг lazy.map.filter не створює проміжних масивів і обчислює лише те, що спожито — виграш на великих даних з first(where:).

Складності: Array — доступ за індексом O(1), insert(at: 0) O(n); Set/Dictionary — contains/lookup O(1) середнє (хешування); Array.contains — O(n).

Яку проблему вирішує

Функціональні операції вирішують проблему «шумних» циклів з мутабельними акумуляторами: менше місць для off-by-one і забутих умов, наміри видно одразу. Знання складностей вирішує проблеми перформансу: перевірка входження в Array у циклі — O(n²), яка на 10 000 елементів вже помітна; заміна на Set — O(n).

Де використовується на практиці
  • Трансформація DTO → доменні моделі: response.items.map(Order.init).
  • compactMap для «безпечного» парсингу: strings.compactMap(URL.init) — невалідні просто відпадають.
  • reduce для агрегації: сума кошика items.reduce(0) { $0 + $1.price * Decimal($1.qty) } — твій CartTotals по суті це робить.
  • Set для швидких перевірок: обрані ресторани favoriteIDs: Set<String> замість масиву.
Приклад
let ids = orders.map(\.id)                       // KeyPath замість closure
let urls = strings.compactMap(URL.init)           // невалідні відпали
let total = cart.reduce(Decimal(0)) { $0 + $1.price }

// lazy: перший підходящий без обробки всього масиву
let first = hugeArray.lazy
    .map(expensiveTransform)
    .first { $0.isValid }
Пастки та типові помилки
  • flatMap на Optional-масивах історично перейменували в compactMap — на співбесіді люблять питати різницю.
  • Ланцюг map.filter.map на великому масиві створює проміжні масиви — lazy або один прохід reduce.
  • Hashable-контракт: рівні обʼєкти зобовʼязані мати рівний hash. Зламаєш (нестабільний hash) — Set/Dictionary поводяться непередбачувано.
  • Мутація колекції під час ітерації — креш/UB; збирай зміни окремо.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • flatMap vs compactMap — різниця та історія?
  • Складність contains у Array vs Set — і коли це реально важливо?
  • Коли lazy-ланцюг виграє, а коли він марний (споживаєш все одно все)?
  • Реалізуй reduce через цикл і суму словника через reduce.

Codable глибоко

🧵 P1
Розгорнути тему
Що це і як працює

Codable = Decodable + Encodable. Компілятор синтезує реалізацію автоматично, якщо всі властивості Codable. CodingKeys — enum відповідності імен властивостей ключам JSON; keyDecodingStrategy = .convertFromSnakeCase — глобальна конвертація. Повний контроль — кастомний init(from decoder:): контейнери (keyed/unkeyed/singleValue), nested containers для вкладених структур.

Яку проблему вирішує

Вирішує ручний парсинг JSON (гори if let json["key"] as? String — багатослівно і криво з помилками). Codable дає декларативний, типобезпечний парсинг з детальними помилками (DecodingError вкаже точний шлях до поля, що не спарсилось).

Де використовується на практиці
  • Кожна модель API у твоєму проєкті. Date через dateDecodingStrategy (.iso8601 або кастомний формуттер — бекенди люблять свої формати).
  • Поле, що буває String або Int (класика реальних бекендів) — кастомний init з послідовними спробами декодування.
  • Стійкість до сміття в масивах: обгортка FailableDecodable, щоб один битий елемент не валив увесь список (критично для стрічок/каталогів).
  • Кеш на диску: JSONEncoder у файл — швидкий офлайн-стан без Core Data.
Приклад
struct Order: Decodable {
    let id: String
    let total: Decimal
    let promoCode: String?          // немає в JSON → nil, без помилки
    enum CodingKeys: String, CodingKey {
        case id = "order_id", total, promoCode = "promo_code"
    }
}
// String або Int в одному полі:
struct FlexibleID: Decodable {
    let value: String
    init(from decoder: Decoder) throws {
        let c = try decoder.singleValueContainer()
        if let s = try? c.decode(String.self) { value = s }

    }

Пастки та типові помилки
  • Оголосив CodingKeys — автосинтез покриває лише перелічені ключі; пропустив властивість без дефолту — помилка компіляції.
  • decodeIfPresent vs decode для optional: decode(String?.self) впаде, якщо КЛЮЧА немає; decodeIfPresent — поверне nil.
  • DecodingError має 4 кейси (typeMismatch, valueNotFound, keyNotFound, dataCorrupted) з context.codingPath — навчись читати, це твій дебаг продакшн-парсингу.
  • Дефолтні значення при відсутньому ключі автосинтез НЕ дає (var x = 0 не допоможе) — потрібен кастомний init або property wrapper.
Контрольні питання — відповів уголос за 1–2 хв → став ✓

  • decode vs decodeIfPresent — різниця?

Init, access control, KeyPath, type erasure

🧵 P2
Розгорнути тему
Що це і як працює

Ініціалізатори класів: designated (повна ініціалізація) і convenience (делегує designated). Two-phase init: фаза 1 — усі stored properties отримали значення знизу вгору по ієрархії; фаза 2 — можна чіпати self. failable init? повертає nil при невалідних даних. required — усі нащадки зобовʼязані реалізувати.

Access control: private (тип+extensions у файлі) < fileprivate (файл) < internal (модуль, дефолт) < public (видно ззовні) < open (ще й можна успадковувати/override ззовні). private(set) — читати можна всім, писати лише всередині.

KeyPath \User.name — типізоване посилання на властивість як значення. Type erasure — приховування конкретного типу за обгорткою (AnyPublisher, AnyView): коли протокол з associated type не можна використати як тип напряму.

Яку проблему вирішує

Two-phase init гарантує, що обʼєкт не використовується напівзібраним (звертання до self до ініціалізації всіх полів — помилка компіляції). Access control — інкапсуляція: private(set) var state дає читання ззовні, а мутацію лише власнику — менше місць, де стан може зламатись. Type erasure вирішує «protocol ... can only be used as a generic constraint».

Де використовується на практиці
  • init?(rawValue:) в enum; init? для валідації (User з порожнім імʼям — nil).
  • private(set) у ViewModel: private(set) var orders: [Order] — View читає, змінює лише VM.
  • KeyPath у API колекцій: map(\.id), sorted(by: \.createdAt) — і в SwiftUI (ForEach(items, id: \.id)).
  • open vs public — актуально для бібліотек/SDK: public клас ззовні НЕ можна сабкласити, open — можна.
Приклад
final class OrderViewModel {
    private(set) var orders: [Order] = []     // ззовні read-only
    private let service: OrderService
    init(service: OrderService) { self.service = service }  // DI через init
}
let names = users.map(\.name)                 // KeyPath
let adults = users.filter { $0.age >= 18 }
Пастки та типові помилки
  • У фазі 1 init класу не можна викликати методи self чи читати властивості — «self used before super.init».
  • convenience init не може ініціалізувати stored properties напряму — тільки делегувати.
  • private у Swift дозволяє доступ з extension того самого типу В ТОМУ Ж файлі — мʼякший за «private» в інших мовах.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Чим open відрізняється від public — і навіщо ця різниця бібліотекам?
  • Що таке two-phase initialization і які помилки він ловить?
  • Що таке type erasure, навіщо AnyView/AnyPublisher? Яку проблему з associated types це обходить?
  • Навіщо private(set) у ViewModel?

Ментальна модель: View як value, identity, rerendering

🧵 P0
Розгорнути тему
Що це і як працює

View у SwiftUI — не обʼєкт на екрані, а дешевий struct-«опис» того, що має бути на екрані. Фреймворк викликає body, порівнює новий опис зі старим і застосовує лише різницю до реального рендер-дерева (attribute graph). body може викликатись дуже часто — тому він зобовʼязаний бути дешевим і без side effects.

Identity відповідає на питання «це та сама вʼюха, що й у попередньому кадрі, чи нова?». Structural identity — позиція у дереві коду (перша гілка if — одна вʼюха, друга — інша). Explicit identity — задана вручну через .id(value) або ForEach(id:). Поки identity збережена — стан (@State) живе; identity змінилась — стара вʼюха знищується разом зі станом, анімується перехід.

Яку проблему вирішує

Ця модель вирішує головну проблему імперативного UI — розсинхрон стану і екрана (забув оновити лейбл після зміни моделі). У SwiftUI екран — функція від стану: змінився стан → фреймворк сам перерахував опис. Ціна — треба розуміти identity, інакше зʼявляються «магічні» баги: стан скидається, анімації не працюють, скрол стрибає.

Де використовується на практиці
  • if isLoggedIn { HomeView() } else { LoginView() } — дві РІЗНІ identity: перемикання повністю пересоздає вʼюху (з анімацією переходу).
  • myView.opacity(isVisible ? 1 : 0) — identity ЗБЕРЕЖЕНА, стан живе; if isVisible { myView } — identity втрачається разом зі станом.
  • .id(userID) на екрані профілю — явний спосіб СКИНУТИ стан вʼюхи при зміні користувача (перевикористання цього ж прийому — pull-to-refresh скидання).
  • Дебаг зайвих перерендерів: let _ = Self._printChanges() у body покаже, яка залежність тригернула виклик.
Приклад
struct CounterView: View {
    @State private var count = 0
    var body: some View {
        let _ = Self._printChanges()   // дебаг: що викликало body
        VStack {
            Text("Count: " + String(count))
            Button("+") { count += 1 }
        }
    }
}
// .id змінює identity → @State скидається:
CounterView().id(resetToken)   // новий resetToken = новий лічильник з нуля
Пастки та типові помилки
  • Side effects у body (мережевий запит, запис у UserDefaults) — body викликається непередбачувано часто; ефекти лише в .task/.onAppear/діях.
  • Виклик body ≠ перерисовка пікселів: SwiftUI дифить описи і чіпає лише реальні зміни. «body викликається часто» — це нормально, якщо він дешевий.
  • AnyView стирає тип і заважає діфингу — уникай у списках; замість нього @ViewBuilder-гілки.
  • Важкі обчислення в body — виноси у ViewModel/computed поза body або кешуй.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Чому @State скидається при зміні identity і як цим свідомо користуватись через .id()?
  • if/else vs opacity — у чому різниця для identity, стану й анімацій?
  • body викликався — чи означає це перерисовку UIKit/Metal-шару? Що насправді відбувається?
  • Як знайти, чому вʼюха перерендерюється занадто часто?

Data flow: @State, @Binding, @StateObject, @ObservedObject, Observation

🧵 P0
Розгорнути тему
Що це і як працює

@State — джерело правди для локального value-стану вʼюхи; SwiftUI зберігає значення ПОЗА структурою (у своєму сховищі, привʼязаному до identity), тому воно переживає пересоздання struct. @Binding — посилання на чуже джерело правди ($-синтаксис створює Binding зі @State): дитина читає і пише, але не володіє.

@StateObject — вʼюха СТВОРЮЄ і володіє ObservableObject: обʼєкт створюється один раз на життя identity. @ObservedObject — вʼюха лише СПОСТЕРІГАЄ за чужим обʼєктом: якщо батько перерендериться і створить новий обʼєкт — стан загубиться. Правило: створюєш тут → @StateObject; передали ззовні → @ObservedObject.

Observation (iOS 17+): макрос @Observable на класі замінює ObservableObject/@Published. Головна перевага — гранулярне відстеження: вʼюха перерендерюється, лише якщо змінилась властивість, яку вона РЕАЛЬНО читала в body (з ObservableObject — при зміні будь-якого @Published). @Bindable дає біндинги до @Observable-обʼєкта.

Яку проблему вирішує

Це відповідь SwiftUI на питання «де живе стан і хто кого оновлює». Класичний баг, який вирішує @StateObject: ViewModel через @ObservedObject у вʼюсі, яку батько часто перебудовує → VM пересоздається, запити перезапускаються, стан зникає. Observation вирішує проблему зайвих перерендерів великих екранів (один @Published тригерив усіх підписників).

Де використовується на практиці
  • @State — прапорці UI: showsSheet, введений текст, вибраний таб.
  • @Binding — переюзабельні компоненти: кастомний пікер/поле, яке редагує значення власника (твої SwiftUI-компоненти в Eatery Club).
  • @StateObject var vm = CheckoutViewModel(...) на екрані оформлення; дочірнім секціям — @ObservedObject/параметри.
  • @EnvironmentObject/@Environment — наскрізні залежності: сесія користувача, тема, DI-контейнер через дерево без прокидання параметрами через 5 рівнів.
  • @AppStorage("isDarkMode") — налаштування напряму з UserDefaults з автооновленням вʼюхи.
Приклад
@Observable final class CheckoutViewModel {   // iOS 17+
    var items: [CartItem] = []
    var isPlacingOrder = false
    private let service: OrderService
    init(service: OrderService) { self.service = service }
}
struct CheckoutView: View {
    @State private var vm: CheckoutViewModel   // з @Observable — просто @State
    init(service: OrderService) {
        _vm = State(initialValue: CheckoutViewModel(service: service))
    }
    var body: some View {
        List(vm.items) { CartRow(item: $0) }    // читає items → підписка лише на items
    }
}
Пастки та типові помилки
  • Класика: @ObservedObject var vm = MyVM() (створення на місці) — VM пересоздається при кожному перерендері батька. Ліки: @StateObject.
  • @StateObject ініціалізується один раз — параметри, передані пізніше, не оновлять його; для залежностей, що змінюються, продумай init або .id().
  • @EnvironmentObject без .environmentObject(...) вище по дереву — креш у рантаймі.
  • З ObservableObject зміна @Published з фонового потоку — фіолетовий warning; @MainActor на VM вирішує системно.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • @StateObject vs @ObservedObject: хто володіє життєвим циклом? Відтвори класичний баг.
  • Чому @Observable ефективніший за ObservableObject + @Published — що саме відстежується?
  • Де фізично зберігається значення @State, якщо View — struct, який постійно пересоздається?
  • Три способи прокинути стан на 3 рівні вниз — тредофи кожного?

Layout system

🧵 P0
Розгорнути тему
Що це і як працює

Однопрохідний протокол із трьох кроків: (1) батько ПРОПОНУЄ дочірній вʼюсі розмір (proposal); (2) дитина ОБИРАЄ власний розмір (може проігнорувати пропозицію — Text не стиснеться нижче за читабельність без явного дозволу); (3) батько РОЗМІЩУЄ дитину у своїх координатах. Розмір дитини — її рішення; позиція — рішення батька.

.frame(width:height:) — не «сеттер розміру», а вʼюха-обгортка, яка пропонує дитині вказаний розмір і сама займає його. fixedSize() — «стань ідеальним розміром, ігноруй пропозицію» (багатострочний Text перестає обрізатись). layoutPriority — кому зі стеку віддавати простір останнім.

Яку проблему вирішує

Вирішує проблему двонаправлених обмежень AutoLayout (конфлікти, unsatisfiable constraints, дебаг рівнянь) — тут конфліктів не буває за побудовою: протокол завжди завершується якимось розміром. Ціна — треба розуміти переговори «пропозиція-відповідь», інакше несподівані обрізання і розтягування.

Де використовується на практиці
  • .frame(maxWidth: .infinity) — «розтягнись на всю пропозицію» — стандарт для кнопок на всю ширину.
  • GeometryReader — коли потрібні реальні розміри/координати (прогрес-бар у частках ширини, паралакс при скролі). Але він жадібний — займає ВСЕ запропоноване, тому обгортай точково.
  • iOS 17+: containerRelativeFrame (частка контейнера без GeometryReader), onGeometryChange (реакція на зміну розміру) — сучасніші точкові інструменти.
  • Двa лейбли в HStack, місця мало: заголовок .layoutPriority(1) — обріжеться підзаголовок.
Приклад
HStack {
    Text(title).layoutPriority(1)     // отримає простір першим
    Spacer()
    Text(subtitle).lineLimit(1)       // стиснеться перший
}
Text("Довгий текст")
    .frame(width: 50)                 // Text обріжеться: він ОБИРАЄ бути
                                      // читабельним, frame лише пропонує
Button("Оплатити") { }
    .frame(maxWidth: .infinity)       // на всю ширину пропозиції
Пастки та типові помилки
  • GeometryReader усередині рядка списку розтягує рядок — обгортай мінімально або бери onGeometryChange.
  • .frame(width: 50) НЕ гарантує, що Text стане 50pt: дитина може обрати інше (обрізання з «...»).
  • Порядок модифікаторів = вкладеність вʼюх: .padding().background(...) ≠ .background(...).padding().
  • Spacer у ScrollView не працює як очікуєш: пропозиція по осі скролу нескінченна.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Опиши три кроки layout-протоколу. Хто вирішує розмір, хто — позицію?
  • Що станеться з Text при .frame(width: 50), якщо текст ширший — і чому?
  • Чому GeometryReader «жадібний» і які iOS 17-альтернативи?
  • Чому padding-background і background-padding дають різні результати?

Lists, ForEach, lazy-контейнери, performance

🧵 P1
Розгорнути тему
Що це і як працює

List — готовий скрол-список з reuse (під капотом UICollectionView), стилями, swipe actions, секціями. LazyVStack у ScrollView — ліниве створення рядків без стилів List (повний контроль верстки), але БЕЗ reuse у класичному сенсі. VStack — створює ВСІ дочірні вʼюхи одразу — тільки для малих наборів.

ForEach — не цикл, а вʼюха-фабрика, яка вимагає стабільної identity елементів (Identifiable або id:). За identity SwiftUI розуміє: цей елемент пересунувся (анімація), цей зник (видалення), цей новий (вставка).

Яку проблему вирішує

Lazy-контейнери вирішують памʼять і час старту: стрічка з 1000 позицій меню створює лише видимі рядки. Стабільні ID вирішують правильність diff-а: з id: \.self на індексах видалення елемента з середини SwiftUI сприймає як «змінились усі елементи після нього» — зламані анімації, загублений стан рядків, стрибки скролу.

Де використовується на практиці
  • Каталог/меню/стрічка замовлень — List або LazyVStack + ForEach(items) (Order: Identifiable за server id — стабільний ID).
  • Пагінація: .task(id: items.last?.id) або .onAppear останнього рядка → підвантаження сторінки.
  • Горизонтальні карусельки (промо-банери в home page): ScrollView(.horizontal) + LazyHStack.
  • Async-картинки у рядках: Kingfisher/AsyncImage з плейсхолдером — не вантаж у body синхронно.
Приклад
struct MenuList: View {
    let items: [MenuItem]                  // MenuItem: Identifiable (id з бекенда)
    var body: some View {
        List(items) { item in
            MenuRow(item: item)
                .onAppear {
                    if item.id == items.last?.id { loadNextPage() }
                }
        }
        .listStyle(.plain)
    }
}
// НЕ РОБИ: ForEach(0..<items.count, id: \.self) — індекси нестабільні
Пастки та типові помилки
  • ForEach(0..<count) з мутабельним масивом — креш або зламаний diff при зміні розміру; діапазонна форма лише для констант.
  • id: \.self на елементах, що повторюються (два однакові рядки) — конфлікт identity, «дивні» анімації.

  • AnyView у рядках списку руйнує оптимізації диффінгу.

Navigation

🧵 P1
Розгорнути тему
Що це і як працює

NavigationStack (iOS 16+) — стек з навігацією за ДАНИМИ: NavigationLink(value: order) кладе значення, navigationDestination(for: Order.self) каже, який екран для цього типу. NavigationPath — типостертий масив-шлях: маніпулюєш ним программно (push/pop/popToRoot = append/removeLast/removeAll).

Модальні: .sheet(isPresented:)/.sheet(item:) — item-варіант звʼязує показ із даними (nil = закрито, значення = показано з цими даними) і прибирає розсинхрон «прапорець true, а даних ще нема». fullScreenCover, presentationDetents (половинні шторки).

Яку проблему вирішує

NavigationStack вирішує головний біль NavigationView: неможливість програмної навігації на N рівнів і крихкі isActive-біндинги. Навігація за даними означає: стан навігації — це просто значення, його можна зберегти, відновити, побудувати з дип-лінка.

Де використовується на практиці
  • Дип-лінк «відкрий замовлення №42 з пуша»: розпарсив URL → path.append(OrderRoute.details(id: 42)) — екран відкрився програмно (твій робочий кейс з universal links).
  • Checkout-флоу: enum CheckoutStep у path — кроки оформлення як дані (сумісно з твоїм server-driven checkout).
  • sheet(item: $selectedProduct) — картка товару: вибір елемента списку відкриває шторку з ним.
  • Роутер-обʼєкт: клас Router з @Published var path — інʼєктиться через environment, будь-який екран може навігувати без звʼязності.
Приклад
enum Route: Hashable {
    case orderDetails(id: String)
    case restaurant(id: String)
}
struct RootView: View {
    @State private var path = NavigationPath()
    var body: some View {
        NavigationStack(path: $path) {
            HomeView()
                .navigationDestination(for: Route.self) { route in

        }
        .onOpenURL { url in
            if let route = DeepLinkParser.parse(url) { path.append(route) }
        }
    }
}
Пастки та типові помилки
  • navigationDestination усередині lazy-контейнера (рядка List) — може не зареєструватись; вішай на контейнер/зовнішню вʼюху.
  • sheet(isPresented:) + окрема опціональна властивість — гонка «шторка відкрилась, дані nil»; sheet(item:) вирішує атомарно.
  • NavigationPath типостертий — «прочитати» маршрути назад не можна; потрібен інтроспект — тримай свій [Route] замість NavigationPath.
  • NavigationView deprecated: дві колонки на iPad несподівано, isActive-хаки — на співбесіді згадай, що мігрував би на Stack.
Контрольні питання — відповів уголос за 1–2 хв → став ✓

Animations і transitions

🧵 P1
Розгорнути тему
Що це і як працює

Анімація у SwiftUI — інтерполяція між старим і новим значенням стану, яку фреймворк робить сам. Implicit: .animation(.spring, value: count) — анімувати зміни, спричинені зміною value. Explicit: withAnimation { state = new } — анімувати все, що змінилось від цієї мутації. Transition — як вʼюха ЗʼЯВЛЯЄТЬСЯ/ЗНИКАЄ (зміна identity): .opacity, .move(edge:), asymmetric.

matchedGeometryEffect — «hero»-анімація: два різні вʼюхи з одним id у одному namespace морфяться одна в одну (мініатюра → повноекранна картка). iOS 17: phaseAnimator (циклічні фази), keyframeAnimator (таймлайн).

Яку проблему вирішує

Вирішує ручне керування анімаціями UIKit (UIView.animate з узгодженням станів, скасуваннями): у SwiftUI анімації interruptible і retargetable з коробки — нове значення посеред анімації плавно перенаправляє поточну, без стрибків.

Де використовується на практиці
  • Розгортання картки товару, показ/приховання фільтрів — withAnimation + transition.
  • Стан кнопки «Оплатити» → лоадер: transition(.opacity.combined(with: .scale)).
  • Промо-банери: matchedGeometryEffect для переходу мініатюра → деталі.
  • Скелетони завантаження: phaseAnimator для пульсації (твій SkeletonView-аналог нативно).
Приклад
struct ExpandableCard: View {
    @State private var expanded = false
    var body: some View {
        VStack {
            header.onTapGesture {
                withAnimation(.spring(duration: 0.35)) { expanded.toggle() }
            }
            if expanded {
                details.transition(.move(edge: .top).combined(with: .opacity))
            }
        }
    }
}
Пастки та типові помилки
  • transition не працює без анімації: зміна умови має бути в withAnimation (або .animation з value) — інакше просто зникне.
  • transition спрацьовує лише при зміні IDENTITY (if/else, ForEach вставки) — не при зміні властивостей.
  • .animation(.spring) БЕЗ value — deprecated: анімував усе підряд, включно з layout при повороті; завжди вказуй value.
  • Анімація в List/скролі може конфліктувати з анімаціями системи — тестуй на пристрої.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • transition vs animation — що анімує кожен і коли спрацьовує transition?
  • Чому .animation без value був deprecated?
  • Як зробити hero-перехід між списком і деталями?
  • Що означає interruptible/retargetable і чим це краще за UIView.animate?

Life cycle і side effects: task, onAppear, App protocol

🧵 P1
Розгорнути тему
Що це і як працює

Точка входу — struct з @main: App → Scene (WindowGroup) → кореневий View. @Environment(\.scenePhase) — active/inactive/background — реакція на згортання (аналог sceneDidEnterBackground).

.task { } — асинхронна робота, привʼязана до життя вʼюхи: стартує при появі, АВТОМАТИЧНО СКАСОВУЄТЬСЯ при зникненні. .task(id:) — перезапуск при зміні значення (новий пошуковий запит при зміні тексту). .onAppear — синхронний хук появи; може викликатись кілька разів (повернення назад по навігації).

Яку проблему вирішує

.task вирішує дві проблеми onAppear + Task {}: (1) витік роботи — запит продовжується після закриття екрана (а .task скасується сам); (2) повторні запити при кожній появі — у .task зручно guard-ити стан. UIApplicationDelegateAdaptor вирішує інтеграцію: пуші, Firebase, deep links вимагають AppDelegate — його підключають у SwiftUI-app без переписування.

Де використовується на практиці
  • .task { await vm.loadMenu() } на екрані меню — завантаження з автоскасуванням.
  • .task(id: searchText) з debounce — живий пошук: новий текст скасовує попередній запит (try await Task.sleep + перевірка).
  • scenePhase == .background → зберегти чернетку кошика.
  • @UIApplicationDelegateAdaptor(AppDelegate.self) — реєстрація пушів, конфіг Firebase (твій робочий сетап).
Приклад
struct SearchView: View {
    @State private var query = ""
    @State private var results: [Restaurant] = []
    var body: some View {
        List(results) { RestaurantRow(restaurant: $0) }
            .searchable(text: $query)
            .task(id: query) {                         // перезапуск на кожну зміну
                try? await Task.sleep(for: .milliseconds(300))  // debounce
                guard !Task.isCancelled, !query.isEmpty else { return }
                results = (try? await api.search(query)) ?? []
            }
    }
}
Пастки та типові помилки
  • onAppear НЕ аналог viewDidLoad: викликається при КОЖНІЙ появі (повернення зі стека) — одноразову ініціалізацію guard-и прапорцем або роби в init VM.
  • Task { } всередині onAppear не скасовується сам — витік роботи; .task робить це за тебе.
  • На iPad/багатовіконність onAppear/onDisappear можуть приходити несподівано — не привʼязуй критичну логіку сесії тільки до них.
Контрольні питання — відповів уголос за 1–2 хв → став ✓

  • Як зробити debounce живого пошуку через .task(id:)?

Розгорнути тему
Що це і як працює

UIViewRepresentable / UIViewControllerRepresentable — обгортка UIKit у SwiftUI. makeUIView — створити (один раз на identity); updateUIView — синхронізувати зі SwiftUI-станом (викликається на кожне оновлення — має бути ідемпотентним і дешевим); makeCoordinator — створити обʼєкт-міст для делегатів/target-action (Coordinator живе стільки ж, скільки representable, і зазвичай тримає звʼязок «UIKit-подія → SwiftUI Binding»).

Вирішує головну проблему реальних проєктів — поступову міграцію: не переписувати десятки UIKit-екранів разом, а додавати нові екрани у SwiftUI всередину наявної навігації (твоя щоденна робота в Eatery Club). І навпаки: використовувати те, чого у SwiftUI немає або воно слабше (карти зі складною поведінкою, WKWebView, кастомні контроли).

Де використовується на практиці
  • Нові екрани у SwiftUI пушаться в наявний UINavigationController через UIHostingController — твій робочий патерн.
  • MapLibre/Google Maps у SwiftUI-екрані — UIViewRepresentable з Coordinator-делегатом карти.
  • UITextField/UITextView зі специфічною поведінкою (маски вводу, тулбари) — representable з Binding тексту.
  • sizingOptions/розміри hosting controller — щоб SwiftUI-контент правильно самовимірювався у клітинках UIKit (UIHostingConfiguration для клітинок iOS 16+).

Архітектура у SwiftUI: MVVM, DI, тестованість

🧵 P1
Розгорнути тему
Що це і як працює

Розподіл: View — лише декларація UI і привʼязка до стану (без бізнес-логіки в body); ViewModel (@Observable/@MainActor) — стан екрана + логіка + виклики сервісів; Services — мережа, сховище, аналітика за протоколами. Стан екрана — enum (idle/loading/loaded/failed), щоб неможливі комбінації не існували.

DI: залежності приходять через init (явно, тестовано) або через Environment (наскрізні). ViewModel залежить від протоколів сервісів — у тестах підставляється мок без мережі.

Яку проблему вирішує

Вирішує «Massive View»: коли запити, форматування і навігація звалені в body — це нетестовано і нечитабельно. MVVM робить логіку тестованою БЕЗ UI: тест створює VM з моком, викликає load(), перевіряє стан. Це прямий шлях закрити твій геп у тестуванні — покрий 2-3 своїх VM юніт-тестами і зможеш предметно про це говорити.

Де використовується на практиці
  • Кожен новий SwiftUI-екран у твоєму проєкті: CheckoutViewModel зі станом, протокол OrderService, мок у тестах.
  • Environment для DI-контейнера: сесія, тема, фіче-флаги (ривок до твого server-driven UI — конфіг як залежність).
  • TCA — знати ідеї оглядово: єдиний стор, редюсери, ефекти; НЕ обовʼязково для співбесід твого рівня, але згадка «знаю концепцію» — плюс.

App lifecycle: AppDelegate, SceneDelegate, стани

🧵 P0
Розгорнути тему
Що це і як працює

Стани застосунку: not running → inactive (працює, не отримує події — дзвінок, шторка) → active → background (обмежений час на завершення справ) → suspended (у памʼяті, не виконується; система може вбити без попередження). Повернення з suspended — НЕ новий запуск, стан живий.

AppDelegate — події рівня ПРОЦЕСУ: запуск (didFinishLaunching — конфіг Firebase, DI, пуші), токен пушів, background fetch. SceneDelegate — події рівня ВІКНА/UI (для multi-window на iPad): створення window, sceneDidBecomeActive, sceneDidEnterBackground. До iOS 13 усе було в AppDelegate.

Яку проблему вирішує

Розуміння станів вирішує реальні продакшн-проблеми: де оновити застарілі дані після повернення з фону (willEnterForeground, не didFinishLaunching — його не буде при поверненні з suspended!), де зберегти стан перед смертю (didEnterBackground — гарантований момент), чому таймери/запити «зупинились» (background = виконання заморожене).

Де використовується на практиці
  • didFinishLaunching: FirebaseApp.configure(), реєстрація пушів, старт DI-контейнера — твій робочий сетап.
  • Пуш-флоу: registerForRemoteNotifications → didRegisterForRemoteNotificationsWithDeviceToken → токен на бекенд; willPresent (пуш прийшов у foreground — показати банер?) / didReceive (юзер тапнув — дип-лінк навігація). Твоя щоденна тема.
  • sceneWillEnterForeground: рефреш статусу замовлення, перевірка протухлого токена — юзер повернувся через годину.
  • didEnterBackground: зберегти чернетку кошика, почати background task на дозапис.
  • Live Activities (твоя фіча!): оновлення через ActivityKit працює З ФОНУ через пуші — гарний приклад роботи навколо lifecycle-обмежень.

    UNUserNotificationCenter.current().delegate = self

UIViewController lifecycle

🧵 P0
Розгорнути тему
Що це і як працює

Ланцюг: init → loadView (створює self.view; переозначають лише для повністю кастомного root view) → viewDidLoad (view створено, розміри НЕ фінальні; один раз на життя VC) → viewWillAppear (перед кожною появою) → viewWillLayoutSubviews / viewDidLayoutSubviews (після проходу AutoLayout — тут розміри ВЖЕ валідні; може викликатись багато разів) → viewDidAppear (на екрані, анімація переходу завершена) → viewWillDisappear / viewDidDisappear → deinit.

Containment (child view controllers): addChild(child) → view.addSubview(child.view) → child.didMove(toParent: self) — вбудовування VC у VC (сегменти, вкладки, твої UIHostingController-вставки).

Яку проблему вирішує

Знання «що і де робити» прибирає цілий клас багів: підписки в viewDidLoad без відписки → дублікати обробників після повернення; робота з frame у viewDidLoad → верстка під старий розмір; важке в viewWillAppear → лаг переходу. Правила: одноразова настройка — viewDidLoad; підписки/таймери — willAppear + зняття в didDisappear; усе залежне від розмірів — viewDidLayoutSubviews.

Де використовується на практиці
  • viewDidLoad: addSubview + констрейнти, конфіг таблиці, привʼязка VM.
  • viewWillAppear: оновити дані, які могли змінитись, поки екран був у стеку (повернулись із деталей — список освіжити); почати спостереження.
  • viewDidLayoutSubviews: градієнтний CALayer під розмір вʼюхи (layer не в AutoLayout!), скругления, залежні від фактичної ширини.
  • viewDidAppear: стартувати анімації, запросити дозволи (не лякати юзера до появи екрана), почати відеоплеєр.
Приклад
override func viewDidLoad() {
    super.viewDidLoad()
    setupSubviews()                    // один раз
}
override func viewWillAppear(_ animated: Bool) {
    super.viewWillAppear(animated)
    NotificationCenter.default.addObserver(self, selector: #selector(refresh),
        name: .orderUpdated, object: nil)
}
override func viewDidDisappear(_ animated: Bool) {
    super.viewDidDisappear(animated)
    NotificationCenter.default.removeObserver(self)   // парна відписка
}
override func viewDidLayoutSubviews() {
    super.viewDidLayoutSubviews()
    gradientLayer.frame = headerView.bounds  // розміри вже фінальні
}
Пастки та типові помилки
  • frame/bounds у viewDidLoad — ще від XIB/дефолту, НЕ фінальні (інші пристрої, rotation). Валідні — у viewDidLayoutSubviews.
  • viewWillAppear без парного «зняття» у didDisappear — подвійні підписки після повернення.
  • viewWillAppear викликався, а viewDidAppear ні — можливо: interactive pop скасовано на півдорозі (свайп назад відпущено).
  • Пропущений didMove(toParent:) при containment — зламані appearance-колбеки дитини.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Повний порядок колбеків від init до deinit? Що викликається один раз, що — багато?
  • Чому frame невалідний у viewDidLoad і де валідний?
  • Що робити в viewWillAppear vs viewDidLoad — правило?
  • Як правильно вбудувати child VC (три кроки) — і де ти це робиш із UIHostingController?

AutoLayout

🧵 P0
Розгорнути тему
Що це і як працює

Система лінійних рівнянь: кожен констрейнт — item1.attr = m × item2.attr + c. Solver (Cassowary) знаходить розвʼязок. Пріоритети: required (1000) — рівняння зобовʼязане виконатись; нижчі (999...1) — «побажання», яким можна поступитись. Конфлікт двох required — креш-лог «Unable to simultaneously satisfy constraints» і система викидає один констрейнт навмання.

Intrinsic content size — власний розмір контенту (лейбл знає розмір тексту). Довкола нього два невидимі констрейнти з пріоритетами: content hugging (опір розтягуванню — «не хочу бути більшим за контент») і compression resistance (опір стисканню — «не дам себе обрізати»). Дефолти: hugging 251, resistance 750.

Цикл верстки: setNeedsLayout — «поміть брудним, перерахуєш на наступному кадрі» (дешево, батчиться); layoutIfNeeded — «перерахуй НЕГАЙНО, якщо брудний» (синхронно); потрібен для анімації констрейнтів.

Яку проблему вирішує

Вирішує адаптивність: один екран на всі розміри iPhone/iPad, динамічні шрифти, локалізації різної довжини (твої 10+ мов — німецький текст в 1.5 раза довший за англійський!). Пріоритети і hugging/resistance вирішують конфлікти «кому поступитись місцем» декларативно.

Де використовується на практиці
  • Класика «два лейбли в рядку»: назва страви + ціна. Ціна не має обрізатись: price.setContentCompressionResistancePriority(.required, for: .horizontal) — стиснеться назва (з «...»).
  • Анімація розкриття панелі: змінити constant → UIView.animate { self.view.layoutIfNeeded() }.
  • Self-sizing клітинки: констрейнти від верху до низу контенту + estimatedRowHeight — твоя щоденна практика в XIB.
  • Перемикання станів: активація/деактивація масивів констрейнтів (NSLayoutConstraint.activate) — компактний/розгорнутий режими.
Приклад
// Два лейбли: ціна не стискається, назва — так
nameLabel.setContentCompressionResistancePriority(.defaultLow, for: .horizontal)
priceLabel.setContentCompressionResistancePriority(.required, for: .horizontal)
priceLabel.setContentHuggingPriority(.required, for: .horizontal)

// Анімація констрейнта:
heightConstraint.constant = isExpanded ? 300 : 80
UIView.animate(withDuration: 0.3) {
    self.view.layoutIfNeeded()   // solver перерахує, зміни анімуються
}
Пастки та типові помилки
  • translatesAutoresizingMaskIntoConstraints = false для КОЖНОЇ вʼюхи, створеної кодом — інакше автогенеровані констрейнти конфліктують з твоїми (найчастіша помилка).
  • Анімація констрейнтів без layoutIfNeeded усередині animate-блоку — зміна стрибне без анімації.
  • Конфлікт 1000 vs 1000 → лог з «Will attempt to recover by breaking constraint» — читай, який зламали; один з required зроби 999.
  • Дебаг: view.constraintsAffectingLayout(for: .vertical), symbolic breakpoint UIViewAlertForUnsatisfiableConstraints, Debug View Hierarchy.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Два лейбли в рядку, місця мало — який стиснеться? Як гарантувати, що ціна ніколи не обріжеться?
  • hugging vs compression resistance — що кожен контролює?
  • setNeedsLayout vs layoutIfNeeded — різниця і чому layoutIfNeeded потрібен в анімації?
  • Як дебажиш Unable to satisfy constraints у великому XIB — покроково?

UITableView / UICollectionView

🧵 P0
Розгорнути тему
Що це і як працює

Reuse: система тримає пул клітинок ≈ під видиму область; клітинка, що поїхала за екран, повертається в пул і видається знову через dequeueReusableCell — зі СТАРИМ вмістом. prepareForReuse — місце скинути стан (скасувати завантаження картинки, обнулити прапорці). dataSource — «що показувати» (кількість, клітинки); delegate — «як поводитись» (висоти, тапи, will/didDisplay).

Diffable Data Source: описуєш НОВИЙ стан снапшотом (секції + item identifiers), система сама рахує diff і анімує вставки/видалення/переміщення. Прибирає цілий клас крешів «number of rows before update is not equal…» від ручного performBatchUpdates. Compositional Layout — декларативний опис колекції: item → group → section, кожна секція зі своєю геометрією й ортогональним скролом.

Яку проблему вирішує

Reuse вирішує памʼять: стрічка з 10 000 замовлень тримає ~15 клітинок замість 10 000. Diffable вирішує коректність оновлень (креші batch updates). Compositional вирішує складні макети (горизонтальні карусельки в вертикальній стрічці — твоя home page) без вкладених collection views і їх багів.

Де використовується на практиці
  • Меню ресторану, список замовлень, каталог — базовий хліб твого проєкту.
  • Async-картинки страв + reuse: скасування завантаження в prepareForReuse (Kingfisher робить це через cancelDownloadTask).
  • Home page із секціями «банери / категорії / рекомендації» — compositional layout з orthogonal scrolling (природно лягає на твій server-driven UI: секції з конфіга бекенда).
  • Diffable + server-driven: прийшов новий конфіг сторінки → новий снапшот → секції красиво анімуються.

// Класичний фікс «чужих картинок» при reuse:
final class DishCell: UITableViewCell {
    override func prepareForReuse() {
        super.prepareForReuse()
        thumbView.kf.cancelDownloadTask()   // 1) скасувати стару загрузку
        thumbView.image = placeholder       // 2) плейсхолдер одразу
    }
}
// 3) у cellForRowAt — завантаження за URL САМЕ ЦІЄЇ моделі

// Diffable:
var snapshot = NSDiffableDataSourceSnapshot<Section, MenuItem.ID>()
snapshot.appendSections([.promo, .dishes])
snapshot.appendItems(promoIDs, toSection: .promo)
dataSource.apply(snapshot, animatingDifferences: true)
Пастки та типові помилки
  • «Чужі» картинки при скролі — питання №1 по таблицях: причина — reuse + асинхронне завантаження, що завершилось для вже переюзаної клітинки. Фікс: три кроки в коді вище.
  • reloadData посеред анімацій/скролу — смикання, втрата виділень; точкові оновлення або diffable.
  • Важке в cellForRowAt (форматування, підрахунки) — виконується на кожен показ клітинки → лаги скролу; готуй дані заздалегідь у VM.
  • Diffable вимагає УНІКАЛЬНИХ і стабільних identifiers; дублікати = креш. ID, а не цілі моделі, у снапшоті (швидше, менше памʼяті).
  • Self-sizing: повний ланцюг вертикальних констрейнтів у contentView + estimatedRowHeight; розриви = нульова висота.

UIView, CALayer, rendering

🧵 P1
Розгорнути тему
Що це і як працює

Кожен UIView — обгортка над CALayer: view обробляє події (touch, responder chain) і AutoLayout, layer — малювання, геометрію, анімації. frame — прямокутник у координатах СУПЕРВʼЮХИ; bounds — власна система координат (origin зазвичай 0,0). Не збігаються: при transform (frame стає undefined — не читай його), при скролі (UIScrollView просто рухає bounds.origin!).

Кадр рендериться конвеєром: layout (layoutSubviews) → display (draw/вміст layer) → композиція на GPU. Offscreen rendering — GPU змушений малювати у проміжний буфер до екрана: тригери — masksToBounds + cornerRadius на складному вмісті, тінь БЕЗ shadowPath, маски. Багато offscreen у клітинках = просідання скролу.

Яку проблему вирішує

Розуміння view/layer пояснює, чому тінь і скруглення «конфліктують» (тінь малюється ЗОВНІ, clipping ріже ВСЕРЕДИНІ — на одному layer несумісні) і як анімувати те, чого немає в UIView API (CABasicAnimation по шляху). Розуміння bounds пояснює весь UIScrollView одним реченням.

Де використовується на практиці
  • Картка страви з тінню і скругленням: два шари — зовнішній layer з shadow + shadowPath, внутрішній view з cornerRadius + masksToBounds.
  • Градієнт кнопки: CAGradientLayer, frame оновлюється у layoutSubviews/viewDidLayoutSubviews (layer не в AutoLayout!).
  • Дебаг скролу: Simulator → Debug → Color Offscreen-Rendered Yellow; Instruments → Core Animation.
  • layoutSubviews викликається при зміні bounds, addSubview, setNeedsLayout — ручна верстка підвʼюх без констрейнтів.
Приклад
// Тінь + скруглення БЕЗ offscreen rendering:
containerView.layer.shadowColor = UIColor.black.cgColor
containerView.layer.shadowOpacity = 0.15
containerView.layer.shadowPath = UIBezierPath(   // явний шлях = без offscreen
    roundedRect: containerView.bounds, cornerRadius: 16).cgPath

contentView.layer.cornerRadius = 16
contentView.layer.masksToBounds = true           // ріже ВНУТРІШНІЙ вміст
// containerView НЕ має masksToBounds — тінь живе зовні
Пастки та типові помилки
  • shadowPath оновлюй при зміні розмірів (layoutSubviews) — інакше тінь відстане від вʼюхи.
  • Після transform читати/писати frame не можна — працюй з bounds + center.
  • drawRect дорогий (CPU-растеризація) — лише коли справді потрібне кастомне малювання; частіше вистачає шарів і картинок.
  • UIStackView сам не малюється (non-rendering) — background до iOS 14 не працював.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • frame vs bounds — коли не збігаються? Як через bounds пояснити UIScrollView?
  • Як зробити тінь + cornerRadius без offscreen rendering?
  • Що тригерить layoutSubviews?
  • Чим відрізняються view і layer за відповідальністю?

Responder chain, hit testing, gestures

🧵 P1
Розгорнути тему
Що це і як працює

Два окремі механізми. Hit testing — ПОШУК вʼюхи під пальцем: рекурсивний обхід дерева ЗВЕРХУ вниз (subviews у зворотному порядку — верхні перші) через hitTest(_:with:), який викликає point(inside:). Вʼюха випадає з пошуку: isUserInteractionEnabled=false, isHidden, alpha<0.01, точка поза bounds.

Responder chain — ДОСТАВКА події, якщо ніхто не обробив: view → superview → ... → UIViewController → UIWindow → UIApplication. UIGestureRecognizer-и мають пріоритет над raw touches; конфлікти вирішуються делегатом (shouldRecognizeSimultaneouslyWith, require(toFail:)).

Яку проблему вирішує

Пояснює найчастіші «містичні» баги дотиків: кнопка, що стирчить за межі батька, НЕ тапається (hit test батька відсікає точку поза його bounds до перевірки дітей); прозорий оверлей зʼїдає тапи; жест таблиці конфліктує з жестом клітинки.

Де використовується на практиці
  • Збільшення тач-зони маленької кнопки: override point(inside:) з bounds.insetBy(dx: -12, dy: -12).
  • Кнопка за межами батька: override hitTest у батька — перевіряти дітей до відсікання.
  • «Пропускати» тапи крізь оверлей: hitTest повертає nil для себе, але дітей перевіряє.
  • Тап по фону закриває клавіатуру, але не блокує кнопки: tap recognizer з cancelsTouchesInView = false.
Приклад
// Збільшена тач-зона:
final class BigTapButton: UIButton {
    override func point(inside point: CGPoint, with event: UIEvent?) -> Bool {
        bounds.insetBy(dx: -12, dy: -12).contains(point)
    }
}
// Кнопка, що виступає за межі контейнера:
override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
    if let v = super.hitTest(point, with: event) { return v }
    let p = overflowButton.convert(point, from: self)
    return overflowButton.point(inside: p, with: event) ? overflowButton : nil
}
Пастки та типові помилки
  • Порядок обходу: ОСТАННЯ додана subview перевіряється ПЕРШОЮ — верхня вʼюха перехоплює тапи нижньої.
  • alpha < 0.01 вимикає дотики — «майже невидима» кнопка перестає працювати.
  • Жест на superview зʼїдає тапи по клітинках таблиці — delegate + touch.view перевірка.
  • hitTest викликається двічі на дотик — не клади туди side effects.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Кнопка стирчить за межі батьківської вʼюхи — чому не тапається і два способи фіксу?
  • Опиши рекурсію hitTest: порядок, умови відсікання.
  • Hit testing vs responder chain — що шукає, що доставляє?
  • Як розвʼязати конфлікт двох жестів (скрол + свайп клітинки)?

Navigation: push, present, координатори

🧵 P1
Розгорнути тему
Що це і як працює

UINavigationController — стек VC: push/pop, свайп назад безкоштовно, спільний navigation bar. present — модальний показ ПОВЕРХ (окремий флоу: оформлення, авторизація); стилі: pageSheet (шторка), fullScreen (важливо: fullScreen НЕ викликає viewWillAppear у батька після dismiss — сюрприз для оновлення даних), popover.

Coordinator: обʼєкт, що ВОЛОДІЄ навігацією — створює VC, тримає navigationController, вирішує «куди далі». VC лише повідомляє про події (делегат/closure) і не знає про інші екрани. Дерево координаторів: AppCoordinator → TabCoordinator → OrderFlowCoordinator.

Яку проблему вирішує

Вирішує зашитість флоу у VC: коли екран сам створює і пушить наступний — його неможливо перевикористати в іншому флоу, тестувати і міняти порядок кроків. Особливо боляче у твоєму випадку server-driven checkout: порядок кроків приходить з бекенда — отже, вирішувати «куди далі» має координатор за конфігом, а не екран.

Де використовується на практиці
  • Checkout-флоу: CheckoutCoordinator читає конфіг кроків (адреса → оплата → підтвердження) і пушить екрани за ним — твій робочий патерн у чистому вигляді.
  • Передача результату назад: closure var onAddressSelected: ((Address) -> Void)? — простіше за delegate для одного колбека.
  • Deep link з пуша: AppCoordinator розбирає маршрут, вибирає таб, будує стек екранів.
  • UIHostingController усередині координатора — SwiftUI-екрани в тому ж флоу без винятків.
Приклад
final class CheckoutCoordinator {
    private let nav: UINavigationController
    private let steps: [CheckoutStep]          // server-driven конфіг!
    private var index = 0
    func start() { show(steps[0]) }
    private func show(_ step: CheckoutStep) {
        let vc = makeViewController(for: step)
        vc.onCompleted = { [weak self] in self?.next() }   // екран не знає, що далі
        nav.pushViewController(vc, animated: true)
    }
    private func next() {
        index += 1
        index < steps.count ? show(steps[index]) : finishOrder()
    }
}
Пастки та типові помилки
  • Координатори ніхто не тримає — деініт і мертві колбеки; батько тримає дітей у масиві childCoordinators, прибирає по завершенню флоу.
  • onCompleted-closure захоплює координатор, координатор тримає VC... — [weak self] у колбеках координатора.
  • present поверх present — «Attempt to present ... which is already presenting»; презентуй з topmost VC.
  • Passing data: синглтони/глобальний стан замість init injection — нетестовано і звʼязує все з усім.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Чому «VC сам пушить наступний VC» — антипатерн? Три причини.
  • Як екран Б повертає результат екрану А — delegate, closure, async? Тредофи?
  • Спроєктуй координатор для server-driven checkout — як конфіг визначає флоу?
  • push vs present — семантика вибору (не технічна різниця, а КОЛИ що)?

RunLoop, main thread, Timer

🧵 P2
Розгорнути тему
Що це і як працює

RunLoop — цикл обробки подій потоку: чекає події (тач, таймер, DisplayLink) → обробляє → спить. Main RunLoop крутить весь UI. Modes: default — звичайна робота; tracking — під час скролу; common — обʼєднання. Timer, доданий у default mode, ЗАВМИРАЄ під час скролу — RunLoop у tracking mode його не бачить.

Яку проблему вирішує

Пояснює класичний баг «таймер зупиняється при скролі» (каруселька банерів перестає гортатись, поки юзер скролить меню — твій прямий кейс!) і чому «UI лише з main» — не забобон, а наслідок однопотокової моделі RunLoop/UIKit.

Де використовується на практиці
  • Автогортання промо-банерів: RunLoop.main.add(timer, forMode: .common) — працює і під час скролу.
  • CADisplayLink — «таймер кадрів» для власних анімацій (прогрес кільця).
  • Main Thread Checker ловить UI-виклики з фонових потоків (фіолетові варнінги).
Приклад
let timer = Timer(timeInterval: 3, repeats: true) { [weak self] _ in
    self?.scrollToNextBanner()
}
RunLoop.main.add(timer, forMode: .common)   // не завмирає при скролі
Пастки та типові помилки
  • Timer.scheduledTimer додає в default mode — банерна каруселька «зависає» при скролі; додай у .common.
  • Timer з target: self тримає self СИЛЬНО до invalidate — block-based API + [weak self] + invalidate у deinit/didDisappear.
  • Таймер — не точний інструмент: спрацює після поточної ітерації RunLoop; для точних кадрів — CADisplayLink.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Чому таймер зупиняється при скролі й як полагодити?
  • Чому Timer(target: self) створює цикл і як уникнути?
  • Що таке RunLoop mode і навіщо common?

XIB / Storyboard vs код

🧵 P2
Розгорнути тему
Що це і як працює

XIB — XML-опис однієї вʼюхи/екрана, вантажиться в рантаймі (Bundle.loadNibNamed); File's Owner — обʼєкт-власник, до якого чіпляються IBOutlet. Storyboard — багато екранів + segue між ними. Верстка кодом — усе явно, найкращий diff/merge.

Твій проєкт (XIB + програмна верстка, БЕЗ Storyboard) — усвідомлений компроміс: XIB дає візуальний огляд і швидкість для статичних екранів/клітинок, код — контроль для динаміки; відсутність Storyboard прибирає найгірші merge-конфлікти і повільний Interface Builder на великих файлах.

Яку проблему вирішує

Вибір інструмента — типове питання «як влаштований ваш проєкт». Storyboard-и в команді конфліктують на кожному PR (один XML на десятки екранів), погано ревʼюються, стрінгові ідентифікатори segue падають у рантаймі. XIB локалізує конфлікт до одного екрана. Код — єдине джерело правди, але повільніше для простих форм.

Де використовується на практиці
  • Переюзабельна вʼюха з XIB: клітинки меню, банери — registerNib + dequeue, твій щоденний пайплайн.
  • loadNibNamed у init кастомної вʼюхи + прикріплення contentView (owner-патерн).
  • Нові екрани — SwiftUI (у тебе так і є) — XIB лишили legacy, нове пишеться декларативно.
Приклад
// Реєстрація і використання клітинки з XIB:
tableView.register(UINib(nibName: "DishCell", bundle: nil),

let cell = tableView.dequeueReusableCell(withIdentifier: "DishCell",
                                         for: indexPath) as! DishCell
Пастки та типові помилки
  • IBOutlet — сучасна рекомендація strong (weak — спадок часів viewDidUnload); але weak теж працює, головне розуміти чому.
  • Забутий registerNib → креш dequeue; стрінгові ідентифікатори — винось у константи/SwiftGen (у вас він є!).
  • XIB + кастомний init: awakeFromNib викликається ПІСЛЯ звʼязування outlets — конфігуруй там, не в init.
Контрольні питання — відповів уголос за 1–2 хв → став ✓

Networking: URLSession і дизайн мережевого шару

🧵 P0
Розгорнути тему
Що це і як працює

URLSession — HTTP-клієнт системи. Конфігурації: .default (кеш, куки на диску), .ephemeral (усе в памʼяті — приватні режими), .background (завантаження продовжуються після вбивства застосунку — через окремий системний процес). Сучасний API: let (data, response) = try await session.data(for: request); статус-код — у (response as? HTTPURLResponse)?.statusCode, і URLSession НЕ кидає помилку на 404/500 — це твоя робота.

Дизайн шару (твоя робоча тема): Endpoint описує запит (path, method, body, headers) як значення; generic-метод request<T: Decodable> виконує і декодує; централізована обробка: mapping статус-кодів у доменні помилки (enum APIError), refresh токена на 401 з повтором запиту, логування.

Яку проблему вирішує

Централізований типізований шар вирішує: розкидані по екранах URLSession.shared.dataTask з копіпастою обробки помилок; «тихі» баги від незвірених статус-кодів; неможливість глобально додати авторизацію/ретраї/логування. Одна точка входу = одне місце для всіх cross-cutting concerns.

Де використовується на практиці
  • Твій продакшн-шар в Eatery Club — розкажи на співбесіді СВОЄ: типізовані запити, централізовані помилки — це готова сильна відповідь.
  • Скасування пошуку: тримаєш Task попереднього запиту, cancel() перед новим (або .task(id: query) у SwiftUI робить це сам).
  • Паралельні незалежні запити екрана: async let / TaskGroup.
  • URLCache + ETag: безкоштовний HTTP-кеш для каталогу; SSL pinning — захист від MITM (знати концепцію: перевірка сертифіката сервера у delegate).
Приклад
enum APIError: Error {
    case unauthorized, notFound
    case server(code: Int), decoding(Error), transport(Error)
}
func request<T: Decodable>(_ endpoint: Endpoint) async throws -> T {
    let (data, response) = try await session.data(for: endpoint.urlRequest)
    guard let http = response as? HTTPURLResponse else { throw APIError.server(code: -1) }
    switch http.statusCode {
    case 200..<300: break
    case 401: throw APIError.unauthorized      // → refresh token flow
    case 404: throw APIError.notFound
    default:  throw APIError.server(code: http.statusCode)
    }
    do { return try decoder.decode(T.self, from: data) }
    catch { throw APIError.decoding(error) }
}
Пастки та типові помилки
  • URLSession не кидає на 4xx/5xx — помилка транспорту ≠ помилка сервера; хто не перевіряє статус, декодить HTML-сторінку помилки як JSON.
  • Retry на 401: онови токен ОДИН раз і повтори; два запити одночасно отримали 401 — refresh має бути дедуплікований (actor/серіалізація), інакше подвійний refresh забанить сесію.
  • Background session вимагає delegate-API (не async), і система викликає колбеки навіть після перезапуску застосунку.
  • Таймаути: timeoutIntervalForRequest — на тишу між пакетами, не на весь запит.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Намалюй свій мережевий шар з роботи: шлях запиту від виклику до декодованої моделі, де ловляться помилки?
  • Чому URLSession не кидає помилку на 500 і що буде, якщо забути перевірити статус?
  • Як реалізувати refresh токена на 401 без подвійного refresh при паралельних запитах?
  • Як скасувати попередній пошуковий запит при введенні нового символа?

Persistence: UserDefaults, Keychain, Core Data / SwiftData, файли

🧵 P1
Розгорнути тему
Що це і як працює

Чотири інструменти під чотири задачі. UserDefaults — маленькі налаштування (прапорці, вибрана мова); plist-файл, НЕ шифрований, вантажиться цілком у память. Keychain — секрети (токени, паролі): шифрується системою, переживає ВИДАЛЕННЯ застосунку, доступ через Security framework (усі користуються обгортками). Core Data — обʼєктний граф зі звʼязками, запитами, міграціями, lazy-loading (faults); це НЕ «обгортка SQLite», а фреймворк керування графом обʼєктів. Файли/FileManager — блоби: картинки, кеш JSON, документи.

Core Data стек: NSPersistentContainer → viewContext (main, для UI) + background contexts (newBackgroundContext / performBackgroundTask). Правило потоків: NSManagedObject живе У СВОЄМУ контексті — між потоками передається objectID, не обʼєкт. SwiftData — новий шар над тим самим ядром: @Model-макрос, @Query у SwiftUI.

Яку проблему вирішує

Правильний вибір сховища — питання і безпеки, і перформансу: токен у UserDefaults читається з бекапа/джейлбрейка за секунди (провал секʼюріті-ревʼю); список з 10 000 обʼєктів у UserDefaults — секунди на кожен запуск; блокуючий запис Core Data на main — фризи UI.

Де використовується на практиці
  • Твій проєкт: access/refresh токени → Keychain; вибраний ресторан, мова, фіче-флаги онбордингу → UserDefaults; кеш меню для офлайн-старту → JSON-файл або Core Data; фото страв → диск через Kingfisher-кеш.
  • Background import: синк каталогу з бекенда — performBackgroundTask, збереження, main context підтягує зміни через automaticallyMergesChangesFromParent.
  • App Group + UserDefaults(suiteName:) — шаринг стану з віджетом/Live Activity (твоя фіча!).
Приклад
// Токен — тільки Keychain:
func saveToken(_ token: Data) {
    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrAccount as String: "accessToken",
        kSecValueData as String: token,
        kSecAttrAccessible as String: kSecAttrAccessibleAfterFirstUnlock
    ]
    SecItemDelete(query as CFDictionary)      // upsert: видалити старий
    SecItemAdd(query as CFDictionary, nil)
}
// Core Data: масовий запис без блокування UI:
container.performBackgroundTask { context in
    for dto in dtos { _ = MenuItem(context: context, dto: dto) }
    try? context.save()     // main context змерджить зміни автоматично
}
Пастки та типові помилки
  • Токен у UserDefaults — класична причина завалу секʼюріті-питання на співбесіді. Тільки Keychain.
  • Keychain переживає видалення застосунку — на першому запуску після перевстановлення можеш знайти старий токен; чисть за прапорцем першого запуску.
  • NSManagedObject між потоками — креш/UB; передавай objectID.
  • UserDefaults.synchronize() — не потрібен з iOS 12, викликати його — маркер застарілих знань.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Що куди зберігати: токен, налаштування, кеш меню, фото — і чому саме так?
  • Чому токен НЕ можна в UserDefaults — два конкретні ризики?
  • Як записати 1000 обʼєктів у Core Data, не заблокувавши UI?
  • Як шарити дані з віджетом/Live Activity (App Group) — твій робочий досвід?

Тестування: XCTest, моки, DI

🧵 P1
Розгорнути тему
Що це і як працює

Unit-тест перевіряє одиницю логіки в ізоляції: швидкий (мілісекунди), детермінований, без мережі/диска/UI. Структура given-when-then. Async-логіка: await прямо в тесті (func test... async throws). XCTest — основний фреймворк; Swift Testing (нове покоління): @Test, #expect, параметризовані тести — знати, що існує.

Мокання: залежність за протоколом → тестова реалізація з контрольованою відповіддю (stub) і записом викликів (spy). DI через init робить підміну тривіальною. Піраміда: багато unit (логіка VM, парсери, форматери) → менше integration → мінімум UI-тестів (повільні й крихкі).

Яку проблему вирішує

Тести вирішують страх рефакторингу і регресії: у твоєму мультибрендовому проєкті зміна в спільному коді може зламати 5 брендів — тести ловлять це до QA. Для співбесіди: відсутність тестів у резюме — найпомітніший твій геп; вміння написати тест VM з моком мережі закриває питання.

Де використовується на практиці
  • Найвигідніші перші тести у твоєму проєкті: розрахунок кошика (CartTotals — чиста логіка, ідеальний кандидат), парсери server-driven конфігів, форматери адрес/цін.
  • VM з моком: тест успіху, тест помилки мережі, тест порожнього стану — три тести на екран, і вже є що показати.
  • Fixtures: static func fixture(...) з дефолтами — читабельні тестові дані без шуму.
Приклад
final class MockOrderService: OrderService {
    var result: Result<[Order], Error> = .success([])
    private(set) var fetchCallCount = 0          // spy: перевірка взаємодії
    func fetchOrders() async throws -> [Order] {
        fetchCallCount += 1
        return try result.get()
    }
}
final class OrdersViewModelTests: XCTestCase {

        mock.result = .failure(APIError.server(code: 500))

        // then

        XCTAssertEqual(mock.fetchCallCount, 1)
    }

Пастки та типові помилки
  • Тести, що ходять у реальну мережу — повільні, флакі, залежні від бекенда; це НЕ unit-тести.
  • Синглтони в коді = неможливість мокати; DI через init — передумова тестованості (тому питання про тести — це питання про архітектуру).
  • Тестуй поведінку, не імплементацію: перевіряй стан VM, а не «викликався приватний метод».
  • XCTestExpectation досі потрібен для callback-API; для async/await — просто await.
Контрольні питання — відповів уголос за 1–2 хв → став ✓
  • Напиши тест VM: успіх і помилка мережі — з моком через протокол.
  • Stub vs spy vs mock — різниця?
  • Що б ти покрив тестами першим у своєму проєкті і чому (підказка: CartTotals)?
  • Чому синглтони заважають тестуванню?

Архітектури і SOLID — розмовна тема

🧵 P0
Розгорнути тему
Що це і як працює

MVC — Apple-дефолт: VC тримає і view-логіку, і презентаційну → «Massive View Controller». MVVM — презентаційна логіка й стан у ViewModel, який НЕ знає про UIKit/SwiftUI → тестований; природна пара для SwiftUI. MVP — presenter командує пасивною view через протокол. VIPER — + Interactor (бізнес-логіка), Router (навігація), Entity: багато файлів на екран, для великих команд із жорсткими межами. Clean Architecture — шари domain/data/presentation, залежності показують всередину (domain нічого не знає про UI і мережу).

SOLID на iOS-прикладах: S — VC, що і верстає, і ходить у мережу, і парсить — розділити; O — нова секція home page додається новим типом, а не switch-ом у 10 місцях (твій server-driven UI — буквально це); L — сабклас не ламає контракт базового; I — не UITableViewDataSource+Delegate в одному гігантському протоколі, а вузькі ролі; D — VM залежить від OrderService-протоколу, не від конкретного APIClient.

Яку проблему вирішує

Архітектура вирішує масштабування коду командою: де шукати баг, як не зачепити чуже, як тестувати. На співбесіді це перевірка зрілості: чи розумієш ТРЕЙДОФИ (VIPER на 3-екранному застосунку — оверінжиніринг; MVC на 50 екранах — болото), а не «яка архітектура наймодніша».

Де використовується на практиці
  • Твоя робоча відповідь: UIKit-екрани — MVC із винесеними менеджерами (CartTotalsManager, CreateOrderManager — по суті інтерактори), нові SwiftUI-екрани — MVVM; server-driven UI зверху. Сформулюй це ЯК СВОЮ архітектурну розповідь — 3 хвилини, впевнено.
  • Server-driven UI — твоя унікальна тема: конфіг з бекенда (таби, секції, кроки checkout) → фабрики екранів/секцій за типом → одна кодова база на N брендів. Підготуй схему словами: конфіг → парсинг → реєстр фабрик → рендер.
  • Патерни, які реально питають: Singleton (чому обережно: глобальний стан, нетестовність), Observer (NotificationCenter/Combine), Factory (твої фабрики секцій!), Coordinator, Adapter (обгортка сторонніх SDK), Facade (твої менеджери).

Пастки та типові помилки
  • «Яка архітектура найкраща?» — питання-пастка: правильна відповідь через тредофи і розмір команди/проєкту, не назва.
  • Не вивчай VIPER напамʼять — вмій пояснити РОЛІ шарів і навіщо стільки межей.
  • Singleton ≠ зло, але: приховані залежності, глобальний мутабельний стан, нетестовність — тому DI навіть для shared-обʼєктів.
  • Готуй ВЛАСНІ приклади: SOLID без прикладів із свого коду звучить як переказ статті.
